# Phase 0 — Interview Guide (Week 1)

**작성일**: 2026-04-23
**상태**: Draft v0.1
**관련**: `phase-0-poc-protocol.md` §5 (이 문서는 질문 상세 + probing)
**대상**: 인터뷰 진행자 (혼자 또는 2인: primary + notetaker)

**이 문서의 역할**: Q1-Q13 **원질문 + follow-up prob + 노트 템플릿**을 담는다. protocol은 "언제 뭘 한다"이고, 이 문서는 "뭐라고 묻는다"이다.

---

## 0. 사용법

1. **인터뷰 5시간 전**: 동의서 서명 받기 (§1), 녹화 소프트웨어 준비
2. **세션 5분 전**: 타이머 + 노트 템플릿 (§ Appendix A) 준비
3. **세션 중**: 이 문서를 참조하며 진행. Probing은 필요 시에만
4. **세션 직후 30분**: 노트를 정리하여 `interview-notes/<id>-session-1.md`로 저장

**원칙**:
- Leading question 피하기 ("이거 불편하시죠?" ❌)
- Self-report + **관찰 병행**. 말과 행동 불일치가 나오면 행동이 진실
- 3초 침묵 견디기 — 응답자가 스스로 생각 정리할 시간 확보
- 모든 painpoint에 "구체적 예시 하나 주실 수 있나요?" probing

## 2. 질문 블록 (60분 구성)

### 2.1 Background (10분) — Q1~Q3

#### Q1. 최근 6개월간 AI 코딩 도구에서 customization을 얼마나 자주 하나요?

- 선택지: a. 매일 / b. 매주 / c. 매월 / d. 드물게 / e. 한 적 없음
- **Probe**: "customization이란 CLAUDE.md 수정, skill 작성, hook 설정 등을 말합니다. 어떤 것들을 해보셨나요?"
- **경고**: "e. 한 적 없음"이면 참가자 자격 재검토 → 인터뷰 계속하되 POC 초청 제외 가능.

#### Q2. 가장 최근에 작성한 customization은 무엇인가요?

- **중요**: 실제 파일을 **화면에 띄워 보여달라고 요청**. 코드/내용 직접 확인.
- **Probe 1**: "이 파일은 어디에 위치하나요?" (경로 확인)
- **Probe 2**: "이걸 쓸 때 참고한 기존 예시가 있었나요?"
- **Probe 3**: "이 파일을 다시 수정한 적이 있나요?"

#### Q3. 그것을 작성하는 데 얼마나 걸렸나요?

- 질문: 첫 draft 시간 + 재시도 횟수 + 최종 안정까지 총 시간
- **Probe**: "첫 시도에 동작했나요, 아니면 여러 번 고쳤나요?"
- **기록**: 재시도 횟수 (≥3이면 painpoint 신호)

### 2.2 Current Workflow 시연 (15분) — 관찰 중심

**지시문** (녹화 시작):
```
"제가 '모든 SQL은 parameterized 쿼리로만' 이라는 규칙을
당신의 AI 어시스턴트에 추가해달라고 요청했다고 가정해주세요.

실제로 어떻게 작업하시는지 보여주시겠어요?
저는 옆에서 관찰만 하고, 중간에 끼어들지 않겠습니다.
막히시면 평소 하시는 대로 검색/질문해주세요."
```

**관찰 체크리스트** (timestamp 기록):

| 시각 | 행동 |
|-----|-----|
| T+0 | 어느 파일 열었는가? (CLAUDE.md / skill / hook / rule / 다른 곳?) |
| T+? | 어디에 쓸지 결정 (망설임 시간) |
| T+? | 기존 예시 검색/조사 (문서, 다른 파일, 웹 검색?) |
| T+? | 포맷/문법 막힘 (구글링? LLM에 질문?) |
| T+? | 저장/테스트 |
| T+? | 동작 확인 (실제로 동작하는지 검증했는가?) |

**Probing (침묵 시에만)**:
- "지금 어디 보고 계시나요?" (만약 자주 멈추면)
- "평소엔 여기서 무엇을 참고하시나요?" (검색/조사 단계에서)

**금지**: 방법 알려주기, "이거 plasmid로 쉬워질 텐데" 언급.

### 2.3 Pain Points (15분) — Q4~Q8

#### Q4. 방금 작업에서 가장 불편했던 점은?

- **Probe 1**: "구체적으로 어느 순간이 가장 답답했나요?"
- **Probe 2**: "그 부분이 없으면 얼마나 빨라졌을까요?"

#### Q5. 작성한 규칙이 실제로 AI 동작에 반영됐는지 어떻게 확인하시나요?

- **Probe 1**: "체크리스트가 있나요, 아니면 우연히 발견하시나요?"
- **Probe 2**: "반영 안 된 걸 나중에 발견한 적 있나요?" (가장 큰 painpoint 신호)

#### Q6. 규칙을 **되돌리고 싶었던** 경험이 있나요? 있다면 어떻게 하셨나요?

- **Probe 1**: "파일 삭제? git revert? 주석 처리?"
- **Probe 2**: "완전히 깨끗하게 되돌아갔다고 확신하시나요?"
- **중요**: 이 질문이 "cure" 가치 검증의 핵심.

#### Q7. 여러 팀원이 이 규칙을 공유하는 방식은?

- **Probe 1**: "어떻게 배포하시나요 (git commit? PR? 구두 공지?)"
- **Probe 2**: "팀원마다 파일 위치가 다른 경우가 있나요?"
- **Probe 3**: "업데이트 시 어떻게 전파?"
- **주의**: 솔로 개발자에겐 "혼자 작업하시면 이 부분 스킵 가능" 안내.

#### Q8. 실패한 customization을 어떻게 관리하시나요?

- **Probe 1**: "실패를 어떻게 인지하셨나요?"
- **Probe 2**: "폴더 어딘가에 남아 있나요, 아니면 깨끗이 삭제?"
- **Probe 3**: "비슷한 실패를 다시 겪은 적 있나요?"

**평가**: Q5-Q7 응답에서 **구체적 painpoint 3개 이상** 추출되는지. Counter 기록.

### 2.4 Concept Exposure (15분) — Q9~Q12

#### Step A: Plasmid 컨셉 설명 (3분 스크립트 낭독)

```
"지금부터 Plasmid라는 개념을 보여드리겠습니다.

 Plasmid는 당신이 원하는 에이전트 동작을 자연어로 선언하는 파일입니다.
 예를 들어 'OWASP 검사를 커밋 전에 강제'라고 쓰면,
 dhelix가 자동으로 hook + agent + rule을 생성합니다.

 마음에 안 들면 /cure 한 번에 다 지워집니다.
 다시 쓰고 싶으면 plasmid 파일만 수정하면 됩니다.

 비유하자면 세포에 plasmid를 주입하면 새 단백질을 만드는 것처럼,
 dhelix에 plasmid를 넣으면 새 에이전트 동작을 얻습니다."
```

#### Step B: POC 예시 3개 시각 노출 (5분)

- 파일 보여주기: `phase-0-sample-plasmids/01-secure-commit-gate.md`
- "이 한 파일이 아래 같이 컴파일됩니다" — 예상 산출물 요약:
  ```
  Generated:
    - .dhelix/hooks/secure-commit-gate.js
    - .dhelix/rules/secure-commit-gate.md
    - .dhelix/agents/secure-commit-gate.md (optional)
  ```
- `/cure` 결과 설명: "위 3개 파일이 깨끗이 사라지고, plasmid 파일만 남음"

#### Q9. 방금 보신 개념을 간단히 설명해주실 수 있나요? (이해도 체크)

- **목적**: 컨셉이 3분 스크립트로 전달 가능한지 검증
- **Probe**: 이해 불완전 시 다시 설명 (이해도 점수 1-5 기록)
- **실패 신호**: "글쎄요... 잘 모르겠어요" → H2에 부정적 신호

#### Q10. 이것이 당신의 current workflow를 개선할까요? (1-5점)

- 1 = 전혀 아님 / 3 = 중립 / 5 = 크게 개선
- **Probe**: "왜 그 점수를 주셨나요?" (정성 근거 기록)
- **경고**: 5점만 있고 이유가 막연하면 social desirability bias 의심

#### Q11. 당장 쓰고 싶으신가요?

- 응답: yes / no / 조건부
- **Probe (조건부 시)**: "어떤 조건이면 쓰시겠어요?"
- **기록**: 조건을 구체적으로 ("로컬 LLM 지원되면", "팀에서 승인하면" 등)

#### Q12. 어떤 우려가 있으신가요?

- **Probe 1**: "프로덕션에 쓴다면 뭐가 걱정되나요?"
- **Probe 2**: "실패 시 롤백은 믿을만한가요?"
- **Probe 3**: "로컬 LLM으로도 동작할까요?" (Ollama 사용자에게 반드시)

### 2.5 POC Scheduling (5분) — Q13

#### Q13. 다음 주에 실제 plasmid 하나를 직접 작성해보실 수 있나요?

```
"30분 세션입니다. 화면 공유 녹화하면서 당신의 실제 프로젝트에서
 plasmid 하나를 작성해보는 실험입니다.

 우리가 minimal template 하나랑 예시 몇 개 드리고,
 당신이 20분간 작성하시면 됩니다. 막히면 질문 OK.

 Ollama 쓰시는 분은 실제로 recombination 돌리는 것까지 해보려고 합니다.

 참여 가능하시면 언제가 좋으실까요?"
```

**기록**:
- [ ] 참여 의향 (yes / no / 조건부)
- [ ] 선호 시간 2슬롯
- [ ] Ollama 환경 재확인 (있는지 / 모델 크기)
- [ ] 프로젝트 공개 가능 여부 (실제 프로젝트 화면 공유)

## 3. 노트 기록 원칙

### 3.1 실시간 기록

- 원문 그대로 중요 quote 기록 (후속 분석 시 paraphrase보다 유용)
- 숫자 정확히 (시간, 횟수, 점수)
- 행동 기록 (참가자가 한 것, 멈춘 곳, 검색어 등)

### 3.2 금지

- Interpretation 끼워 넣기 ("~를 의미하는 것 같다")
- Leading question 후 응답을 "자발적"으로 기록
- 녹화만 믿고 노트 skip

### 3.3 세션 직후 30분 — 정리

- 원 quote 선별 (5개 이하, painpoint별)
- Q별 응답 + probing 기록 합치기
- 자신의 inference는 별도 섹션 `## Interpretations (interviewer note)` 로 분리

## 5. 자주 나올 질문에 대한 응답 스크립트

**Q: "Cursor나 Continue처럼 rule만 쓰면 되지 않나요?"**
```
"Rule은 하나의 출력물이지만, plasmid는 여러 artifact (rule + hook + agent)를
 동시에 생성하는 상위 개념이에요. 그리고 다 깨끗이 되돌릴 수 있죠.
 그 차이가 의미 있게 느껴지시나요?" (open-ended)
```

**Q: "제가 작성한 게 잘못되면 어떻게 되나요?"**
```
"4단계 validation (L1-L4)이 있어서 실패 시 자동 롤백됩니다.
 단 실제 어떻게 동작할지는 Phase 1+에서 구현할 거고, 지금은 컨셉 검증 단계예요."
```

**Q: "개인정보는 어떻게 되나요?"**
```
"녹화는 내부 팀만 보고 90일 후 삭제됩니다. 노트는 익명화돼요.
 언제든 삭제 요청하실 수 있어요."
```

**Q: "이거 언제 나와요?"**
```
"오늘 검증 통과되면 5주 후 Phase 1 alpha가 나올 거예요.
 베타 사용자로 초대해도 될까요?" (H2 행동 지표!)
```
