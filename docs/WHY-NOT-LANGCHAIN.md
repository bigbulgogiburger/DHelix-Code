# 왜 LangChain을 쓰지 않았는가

> "왜 LangChain 안 썼어요?" — 이 질문에 대한 기술적 답변

---

## TL;DR

LangChain은 **프로토타입용 프레임워크**다. dhelix는 **프로덕션 AI 코딩 에이전트**다.

LangChain의 에이전트 루프 위에서는 우리가 필요로 하는 12가지 핵심 기능 중 **어느 것도 구현할 수 없거나**, 구현하려면 프레임워크를 fork해서 코어를 다시 써야 한다. 그건 "프레임워크를 쓴다"가 아니라 "프레임워크와 싸운다"는 뜻이다.

---

## 1. LangChain이 뭔데?

LangChain은 LLM 애플리케이션을 빠르게 만들기 위한 Python/JS 프레임워크다. Chain, Agent, Tool, Memory 같은 추상화를 제공해서, 몇 줄의 코드로 "AI가 도구를 쓰면서 대화하는" 시스템을 만들 수 있다.

```python
# LangChain으로 에이전트 만들기 (간단해 보인다)
agent = create_react_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools)
result = agent_executor.invoke({"input": "파일을 읽어줘"})
```

**그럼 왜 안 썼냐고?** 아래 12가지 이유를 보자.

---

## 2. LangChain으로는 불가능한 것들

### 2.1 스트리밍 + 도구 호출 동시 처리

LLM은 텍스트와 도구 호출 인자를 **인터리빙**해서 보낸다:

```
텍스트 청크 → 도구 인자 청크 → 텍스트 청크 → 도구 인자 청크 → ...
```

dhelix는 이걸 동시에 처리한다. 사용자는 AI가 글을 쓰는 걸 실시간으로 보면서, 백그라운드에서 도구 호출 인자가 조립된다.

**LangChain의 한계:** 텍스트 스트리밍과 도구 호출을 순차 처리한다. 도구가 바인딩되면 토큰 레벨 스트리밍이 안 되는 [공개 이슈](https://github.com/langchain-ai/langchain/issues/26971)가 존재하고, 스트리밍 중 토큰 사용량 추적도 [불가능](https://github.com/langchain-ai/langchainjs/issues/6583)하다.

### 2.2 서킷 브레이커 — 의미 기반 무한루프 탐지

LangChain은 `max_iterations`으로 하드 리밋만 건다. "50번 반복하면 멈춰라."

dhelix는 **진행 상황을 의미적으로 분석**한다:

```
3번 연속 파일 수정 0건 + 출력 0건 → 정지 (진전 없음)
5번 연속 같은 에러 메시지 → 정지 (같은 실수 반복)
같은 도구를 같은 인자로 반복 호출 → 정지 (루프 감지)
```

이건 단순 카운터가 아니라 **패턴 인식**이다. LangChain의 에이전트 루프에는 이런 시맨틱 분석이 없다.

### 2.3 대화 중 컨텍스트 압축 (3단계)

대화가 길어지면 토큰이 부족해진다. LangChain은 이걸 처리할 메커니즘이 없다.

dhelix는 3단계 전략을 쓴다:

| 단계              | 타이밍     | 동작                                                               |
| ----------------- | ---------- | ------------------------------------------------------------------ |
| **마이크로 압축** | 상시       | 큰 도구 출력(200토큰+)을 디스크로 이동. 최근 5개만 인라인 유지     |
| **자동 압축**     | 83.5% 도달 | 오래된 대화를 구조화된 요약으로 교체 (의도, 결정, 파일, 에러 보존) |
| **재수화**        | 압축 후    | 최근 수정한 파일을 다시 읽어서 LLM에게 최신 상태 제공              |

특히 "콜드 스토리지"는 도구 출력을 content-addressable hash로 디스크에 저장하고, 필요할 때 다시 불러오는 시스템이다. LangChain에는 이런 개념 자체가 없다.

### 2.4 관찰 마스킹 — 재생성 가능한 출력 구분

`file_read`의 결과는 언제든 다시 읽을 수 있다. `grep_search`도 같은 명령을 다시 치면 된다. 이런 건 컨텍스트에 계속 들고 있을 필요가 없다.

dhelix는 도구 출력을 두 종류로 구분한다:

- **재생성 가능** (file_read, grep, glob, 읽기 전용 bash) → 마스킹 대상
- **재생성 불가** (file_write, file_edit, 파괴적 bash) → 절대 마스킹하지 않음

오래된 재생성 가능 출력은 플레이스홀더로 교체: `[Observation masked — file_read (1200 tokens). Re-read if needed.]`

LangChain은 모든 도구 출력을 **불변 이력**으로 취급한다.

### 2.5 적응형 스키마 — 모델 능력에 따른 도구 정의 조절

GPT-4는 20개 파라미터가 있는 복잡한 JSON 스키마를 잘 이해한다. Claude Haiku는 혼란스러워한다.

dhelix는 모델 티어에 따라 **같은 도구의 스키마를 다르게** 보낸다:

| 티어                      | 전략                                    |
| ------------------------- | --------------------------------------- |
| HIGH (GPT-4, Claude Opus) | 전체 스키마, 모든 파라미터, 완전한 설명 |
| MEDIUM (Claude Sonnet)    | 핵심 파라미터만, 설명 2문장으로 축약    |
| LOW (Haiku, 구형 모델)    | 필수 파라미터만 + few-shot 예시 추가    |

LangChain은 도구당 **스키마가 하나**다. 모델에 따라 조절하는 기능이 없다.

### 2.6 인라인 권한 시스템

AI가 `rm -rf /`를 실행하려고 하면? 사용자에게 물어봐야 한다.

dhelix의 권한 체크는 **도구 실행 파이프라인 안에** 내장되어 있다:

```
도구 호출 추출 → 권한 확인 (사용자 응답 대기) → 가드레일 검사 → 실행 → 출력 필터링
```

이 과정에서 `async` 콜백으로 사용자의 승인/거부를 기다리고, 거부된 도구는 건너뛰되 나머지는 병렬 실행한다.

LangChain의 도구 루프는 **중간에 사용자 입력을 받는 메커니즘이 없다**. 도구는 실행되거나 안 되거나 둘 중 하나다.

### 2.7 도구 호출 자동 교정

저가 모델들은 자주 실수한다:

- 상대 경로 전달 (`./src/app.ts` → `/project/src/app.ts`로 교정)
- `"true"` 문자열 전달 (`true` boolean으로 교정)
- `"123"` 문자열 전달 (`123` 숫자로 교정)

dhelix는 Zod 검증 **전에** 이런 실수를 자동으로 고친다. 덕분에 저가 모델의 도구 호출 성공률이 올라간다.

LangChain은 도구 호출이 유효하거나 아니거나 둘 뿐이다. "고쳐서 재시도"가 없다.

### 2.8 파일 충돌 감지 기반 병렬 실행

AI가 한 번에 여러 도구를 호출할 때:

```
file_read("a.ts") + grep_search("foo")     → 병렬 OK (둘 다 읽기)
file_write("a.ts") + file_read("b.ts")     → 병렬 OK (다른 파일)
file_write("a.ts") + file_edit("a.ts")     → 순차 실행! (같은 파일 충돌)
```

dhelix는 도구 호출을 **충돌 감지**를 거쳐 그룹으로 나누고, 그룹 내에서는 `Promise.allSettled`로 병렬 실행, 그룹 간에는 순차 실행한다.

LangChain은 도구 호출 간 **파일 경로 추적이나 동시성 제약 분석**을 하지 않는다.

### 2.9 에러 유형별 복구 전략

```
"request too large"  → 컨텍스트 압축 후 재시도
"timeout"           → 지수 백오프 후 재시도
"JSON parse error"  → 텍스트 파싱 전략으로 전환
"rate limit (429)"  → 대기 후 재시도
```

LangChain의 에러 처리는 이분법적이다: 재시도하거나 실패하거나. **에러 유형에 따라 다른 복구 전략**을 적용하는 메커니즘이 없다.

### 2.10 자동 체크포인트

AI가 파일을 수정하기 **직전에** 해당 파일을 스냅샷한다. 에이전트가 실수하면 `undo`로 되돌릴 수 있다.

```
file_write("app.ts") 실행 전 → app.ts 내용 저장
file_edit("utils.ts") 실행 전 → utils.ts 내용 저장
grep_search 실행 전 → 아무것도 안 함 (읽기 전용)
```

LangChain은 파일 시스템과 통합되어 있지 않으므로, 이런 체크포인트 로직 자체가 구조적으로 불가능하다.

### 2.11 듀얼 모델 라우팅

비싼 모델(GPT-4)로 **설계**하고, 싼 모델(Haiku)로 **실행**하는 패턴:

```
사용자: "인증 시스템 만들어줘"
  → [계획 단계] architect 모델 (GPT-4) 호출 → 설계서 생성
  → [실행 단계] editor 모델 (Haiku) 호출 → 코드 생성
  → [리뷰 단계] architect 모델 (GPT-4) 호출 → 코드 검토
```

dhelix는 대화 흐름을 분석해서 **지금이 계획 단계인지 실행 단계인지** 자동 감지하고, 모델을 바꿔 끼운다.

LangChain은 에이전트당 하나의 모델을 사용한다. 단계별 라우팅 개념이 없다.

### 2.12 인라인 보안 가드레일

도구 실행 전후로 보안 검사가 **파이프라인 안에서** 실행된다:

**실행 전 (입력 가드레일):**

- `rm -rf /` 같은 위험 명령 차단
- `~/.ssh/id_rsa` 같은 민감 경로 접근 차단
- 프롬프트 인젝션 시도 감지

**실행 후 (출력 가드레일):**

- API 키, 비밀번호 등 비밀 정보 스캔
- 고엔트로피 문자열 감지
- 출력 크기 제한

이건 미들웨어가 아니라 **에이전트 루프의 일부**다. LangChain은 이런 검사를 볼트온 미들웨어로 붙여야 하는데, 그러면 루프 내부 상태에 접근할 수 없다.

## 3. 업계에서 LangChain을 비판하는 이유

우리만의 의견이 아니다. 업계 전체가 같은 결론에 도달하고 있다.

### 3.1 창시자 본인도 인정했다

Harrison Chase (LangChain 창시자)가 Latent Space 팟캐스트에서 말했다:

> "사람들은 우리가 제공하는 것보다 **훨씬 더 많은 유연성과 제어권**을 원했다."
>
> "시작하기 쉽게 만든 그 고수준 인터페이스가, 프로덕션 커스터마이징을 하려는 순간 **방해물**이 됐다."

이 문제를 해결하기 위해 LangGraph를 만들었다는 건, **LangChain의 에이전트 루프가 불충분했다**는 것을 스스로 인정한 셈이다.

### 3.2 프로덕션 이탈율

2025년 AI 개발자 설문 결과:

- LangChain을 실험한 개발자의 **45%가 프로덕션에서 사용하지 않음**
- 프로덕션에 도입했던 개발자의 **23%가 나중에 완전히 제거**

### 3.3 Octomind 사례

AI 기반 E2E 테스팅 회사 Octomind는 LangChain을 12개월 이상 프로덕션에서 사용한 뒤 **완전히 제거**했다:

> "LangChain은 생산성의 원천이 아니라 **마찰의 원천**이 됐다."
>
> "프레임워크 없이 에이전트를 만드는 게 가능하고, **자체 도구를 만드는 투자가 그만한 가치가 있다.**"

### 3.4 성능 오버헤드

| 지표           | 직접 API 호출 | LangChain 경유 | 오버헤드 |
| -------------- | ------------- | -------------- | -------- |
| 단순 요청 지연 | 850ms         | 1,020ms        | **+20%** |
| 인스턴스 생성  | N/A           | 150-160ms      | 매번     |

한 핀테크 회사는 LangChain → 커스텀 오케스트레이터 전환 후 **지연 40% 감소, 연간 $200K GPU 비용 절감**.

### 3.5 디버깅의 지옥

Hacker News의 반복되는 피드백:

> "5단계의 추상화를 통과해야 작은 디테일 하나를 바꿀 수 있다."
>
> "팀이 LangChain을 이해하고 디버깅하는 데 쓰는 시간이, 기능을 만드는 시간과 **같아졌다.**"

LangChain에서 오류가 나면 그게 프롬프트 문제인지, 체인 문제인지, 콜백 문제인지, 프레임워크 내부 문제인지 **구분할 수 없다.**

### 3.6 의존성 비대

`langchain-community` 패키지만 **33.2MB**. 간단한 앱이라도 HTTP 클라이언트, ML 도구 등 불필요한 라이브러리가 수십 개 딸려 온다. 서버리스 환경에서는 콜드 스타트 지연과 메모리 사용 증가로 직결된다.

## 4. 비교 요약

| 기능                      | LangChain        | dhelix 커스텀                | 필수?    |
| ------------------------- | ---------------- | ---------------------------- | -------- |
| 스트리밍 + 도구 호출 동시 | 순차만 가능      | 인터리빙 처리                | **필수** |
| 무한루프 감지             | max_iterations만 | 시맨틱 진행 분석             | **필수** |
| 대화 중 컨텍스트 압축     | 없음             | 3단계 콜드 스토리지 + 재수화 | **필수** |
| 관찰 마스킹               | 없음             | 재생성 가능 여부 판별        | **필수** |
| 적응형 도구 스키마        | 고정 스키마      | 모델 티어별 최적화           | **필수** |
| 인라인 권한 시스템        | 없음             | 계층적 권한 + 세션 캐싱      | **필수** |
| 도구 호출 자동 교정       | 없음             | 저가 모델 호환 보정          | 중요     |
| 파일 충돌 기반 병렬화     | 전부 병렬        | 충돌 감지 + 그룹 실행        | **필수** |
| 에러 유형별 복구          | 재시도/실패      | 전략 기반 분기               | **필수** |
| 자동 체크포인트           | 없음             | 파일 변경 직전 스냅샷        | 중요     |
| 듀얼 모델 라우팅          | 없음             | 단계별 자동 전환             | 중요     |
| 인라인 가드레일           | 미들웨어         | 파이프라인 내장              | **필수** |

**12개 중 9개가 "필수"이고, LangChain에서는 지원하지 않거나 불가능하다.**

## 5. "그래도 LangChain을 쓰면 좋았을 텐데..."에 대한 답변

### "빠르게 시작할 수 있잖아"

맞다. 하지만 **시작은 빨라도 도착은 느리다.**

LangChain으로 프로토타입을 2일 만에 만들 수 있다. 하지만 컨텍스트 압축, 권한 시스템, 가드레일을 붙이려는 순간 프레임워크와 싸우기 시작한다. 결국 fork하거나 버린다. LangChain을 프로덕션에 올렸다가 제거한 23%의 팀이 이 과정을 겪었다.

### "LangGraph가 해결했잖아"

LangGraph는 LangChain의 에이전트 루프가 **불충분함을 인정한 결과물**이다. 하지만 LangGraph도 그래프의 노드/엣지 수준에서 동작하므로, 우리가 필요로 하는 파일 시스템 통합, 토큰 레벨 스트리밍, 콜드 스토리지 같은 저수준 제어를 제공하지 않는다.

### "추상화가 미래를 대비해 주잖아"

추상화는 **실제 변경 지점**을 알 때만 가치가 있다. LangChain의 추상화는 LLM 제공자 교체를 쉽게 해주지만, 우리는 이미 OpenAI 호환 API 하나로 GPT-4, Claude, Ollama, vLLM을 모두 지원한다. 추상화 레이어가 하나 더 필요하지 않다.

### "커뮤니티와 에코시스템은?"

dhelix가 필요로 하는 도구 통합은 MCP (Model Context Protocol)로 처리한다. MCP는 LangChain 에코시스템보다 **표준화된 프로토콜**이고, 어떤 프레임워크에도 종속되지 않는다.

---

## 6. 결론

직접 에이전트 루프를 구현한 건 NIH 증후군(Not Invented Here)이 아니다.

OpenAI API로 LLM을 호출하고, 도구 호출을 파싱하고, 결과를 돌려보내는 루프 자체는 **100줄이면 된다.** 프레임워크가 필요할 만큼 복잡한 작업이 아니다.

하지만 그 루프 위에 컨텍스트 압축, 권한 시스템, 보안 가드레일, 서킷 브레이커, 적응형 스키마, 듀얼 모델 라우팅, 자동 체크포인트를 쌓으면 — 이건 **우리만의 도메인 로직**이다. 범용 프레임워크가 대신해줄 수 없다.

> **프레임워크를 쓰는 이유는 반복적인 보일러플레이트를 줄이기 위해서다.** > **프레임워크를 쓰지 않는 이유는 우리의 핵심 로직이 보일러플레이트가 아니기 때문이다.**
