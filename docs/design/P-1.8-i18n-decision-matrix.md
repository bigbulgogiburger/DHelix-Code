## 0. TL;DR

- Schema 식별자 (Zod field names, enum values) = **영어 고정**
- User 자유 텍스트 (description, tags, body) = **자유 (어떤 언어든)**
- Interpreter LLM 프롬프트 = **영어 고정 (LLM 공통어)**
- Generated artifact = **plasmid body 언어 따름**
- CLI error/UX = **CLI locale 따름**

## 1. 레이어별 결정 매트릭스

| 레이어 | 결정 | 근거 | 예시 |
|-------|-----|-----|-----|
| Frontmatter field names (Zod) | 영어 고정 | Schema validation 일관성, Zod 키 타이핑 | `name`, `description`, `scope` |
| Frontmatter enum values | 영어 고정 | Schema validation | `high`, `foundational`, `structural` |
| Frontmatter 자유값 (description, tags) | 자유 | 사용자 자연어 | `description: "OWASP 검사 강제"` (한글 OK) |
| Plasmid body | 자유 | Interpreter가 언어 감지 후 처리 | `## Intent\n모든 커밋에...` |
| Interpreter LLM 프롬프트 | 영어 고정 | LLM 공통어 (영어가 가장 안정) | "Analyze the following plasmid body..." |
| Generated artifact 본문 | Plasmid body 언어 따름 | UX 일관성 (사용자가 한글로 썼으면 artifact도 한글) | agent 설명이 한글이면 agent 프롬프트도 한글 |
| Template comments (`.hbs`) | 영어 고정 | 코드 레이어 | `// Generated by plasmid: ...` |
| Error codes | 영어 고정 | 로그/알림 consistency | `PLASMID_PARSE_ERROR` |
| Error messages (CLI) | CLI locale 따름 | 접근성 | "플라스미드 파싱 실패" (ko) / "Plasmid parse failed" (en) |
| Challenge log rationale | 자유 (사용자 언어) | User-authored | "이유는 ..." |

## 2. 언어 감지 함수

```typescript
// src/recombination/interpreter/language.ts
export type Language = "ko" | "ja" | "zh" | "en" | "other";

export function detectPlasmidLanguage(body: string): Language {
  // 한글 (U+AC00-U+D7AF Hangul Syllables)
  const koreanMatches = body.match(/[가-힯]/g)?.length ?? 0;
  if (koreanMatches / body.length > 0.05) return "ko";

  // 일본어 — 히라가나 (U+3040-U+309F) + 가타카나 (U+30A0-U+30FF)
  const japaneseMatches = body.match(/[぀-ゟ゠-ヿ]/g)?.length ?? 0;
  if (japaneseMatches / body.length > 0.05) return "ja";

  // 중국어 — CJK Unified (U+4E00-U+9FFF, Japanese도 포함이지만 hiragana/katakana 없으면 중국어)
  const cjkMatches = body.match(/[一-鿿]/g)?.length ?? 0;
  if (cjkMatches / body.length > 0.05) return "zh";

  // ASCII-dominant → en
  const asciiRatio = body.match(/[\x00-\x7F]/g)?.length ?? 0;
  if (asciiRatio / body.length > 0.9) return "en";

  return "other";
}
```

### 2.1 한계

- 혼합 언어 (한글 + 영어 코드 샘플) 정확히 감지 어려움 — threshold 5% 로 한쪽 우세하면 그쪽으로 판정
- 프로그래밍 언어만 있는 body는 `en`으로 오판정 — 문제 없음 (artifact도 en)
- 이모지 많은 body — non-ASCII 비율 높지만 언어 판정 불가 → `en` fallback

### 2.2 신뢰도

Phase 0 POC에서 참가자 plasmid body 언어 감지 정확도 측정. 90%+면 충분. 미달이면 `franc` 같은 라이브러리 도입 고려.

## 3. Interpreter 프롬프트 언어 지침

```
System: You analyze plasmid intents for a code agent.

User: Analyze the following plasmid body (written in ${detectedLanguage}).

Output JSON schema fields in English (name, kind, artifact paths, etc.).
Preserve user-facing strings in ${detectedLanguage}: descriptions, rationales, user prompts.

Plasmid body:
---
${body}
---

Output JSON:
{ "artifacts": [...], "compressedSummary": string (in ${detectedLanguage}), ... }
```

**핵심**:
- 스키마 필드 (JSON key) 는 영어
- value 중 **사용자에게 노출되는 문자열**은 body 언어 유지

## 5. CLI Locale

환경 변수 우선순위:

1. `DHELIX_LOCALE=ko`
2. `LANG=ko_KR.UTF-8` → `ko`
3. default: `en`

```typescript
// src/cli/i18n.ts
export function currentLocale(): Locale {
  const explicit = process.env.DHELIX_LOCALE;
  if (explicit) return normalizeLocale(explicit);
  const system = process.env.LANG ?? process.env.LC_ALL ?? "en";
  return normalizeLocale(system);
}
```

## 9. Resolved Questions (v0.2)

### Q1 — 언어 override frontmatter 필드

**확정**: **`language: "ko" | "en" | "ja" | "zh" | "auto"` optional 필드 추가 (default "auto")**.

```typescript
// src/plasmids/schema/v1.ts
language: z.enum(["ko", "en", "ja", "zh", "auto"]).default("auto"),
```

`auto` 는 `detectPlasmidLanguage()` 호출 (§2). 나머지는 override.

**Rationale**: 언어 감지 오작동 (예: 영어 코드 스니펫 많은 한글 plasmid) 사용자 수정 경로 필요. 5 enum 으로 schema validation 유지.

### Q2 — CLI 메시지 missing key 처리

**확정**: **`en` fallback + 첫 1회만 debug log warning**.

```typescript
// src/cli/i18n.ts
export function t(key: string, locale: Locale = currentLocale()): string {
  const value = MESSAGES[locale]?.[key];
  if (value) return value;
  const fallback = MESSAGES["en"][key];
  if (fallback) {
    warnOnce(`i18n missing key "${key}" in locale "${locale}", using en fallback`);
    return fallback;
  }
  warnOnce(`i18n missing key "${key}" in ALL locales, using key itself`);
  return key;   // 최악의 경우 key 자체 반환
}
```

**Rationale**: Silent missing 은 UX 혼란 (빈 문자열). en fallback 은 항상 동작. Key 자체 반환은 dev-time typo 감지.

### Q3 — RTL 언어 지원

**확정**: **v0.4 scope 밖. v0.5+ 로드맵 포함**.

우선순위 낮은 이유:
- CLI terminal 에서 RTL rendering 복잡 (Ink React 라이브러리 제약)
- dhelix 주 사용자는 기술 개발자 — 대부분 en/ko/ja/zh/es 사용
- RTL 요청 발생 시 Phase 6+ polish 에서 재검토

**로드맵 기록**: `docs/roadmap.md` 에 "v0.5+: Arabic/Hebrew plasmid body 지원 (CLI bidi layout + message bundle)".

### Q4 — `ja-zh` 구분 정확도

**확정**: **현행 휴리스틱 (hiragana/katakana 유무) 유지. Problem 발생 시 `franc` 라이브러리 도입**.

**Rationale**:
- 현행 방식 정확도 ~95% (hiragana/katakana 이 실제 일본어 텍스트 거의 항상 포함)
- 일본어 한자만 있는 body (예: 고전 일본어) 는 edge case — POC 참가자 없을 것
- `franc` 은 400KB gzipped dependency — 정확도 대가로는 크지 않지만 불필요한 bloat
- 실제 오판정 신고 발생 시 trigger

**영향**: `docs/known-issues.md` 에 "`ja-zh` 구분 한계" 항목 기록.
