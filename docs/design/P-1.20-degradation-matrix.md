## 6. Fallback Chain 참조

각 stage에서 "앞 단계 실패 시 어디로" 요약:

```
Stage 2a fail (interpret)
  └─ Retry (count per tier)
     └─ XML fallback (cloud + local-large)
        └─ Field-by-field split (local-small)
           └─ Abort + PLASMID_PARSE_ERROR

Stage 2c fail (compression)
  └─ LLM summary timeout / parse error
     └─ Layer A fallback (description only)

Stage 2d fail (reorg)
  └─ JSON parse fail → XML retry (if reliableJson)
     └─ Deterministic fallback (always works)

Stage 6 fail (validation)
  └─ L1 1 fail or L2 threshold → rollback
     └─ Grace period 10s → user override possible
```

## 8. Resolved Questions (v0.2)

### Q1 — Local Small `template-only` artifact 실용성

**확정**: **v0.4 포함. Phase 0 POC Ollama 참가자 실측으로 통과율 측정 + v0.5 조정 여지**.

가정: template-only artifact는 **Handlebars slot이 기본값 또는 빈 문자열** 로 채워진 상태. 이 상태에서 static validation (P-1.3) 통과 + runtime trigger 동작이 목표.

Phase 0 POC 예측 지표:
- Static validation 통과율: 95%+ (기본값 안전하게 설계됨)
- Runtime L1 trigger rate: 60-75% (LLM 없이 기본값으로만)
- 사용자 만족도: "수정 후 사용" 필요성 높음

**v0.5 개선 trigger**: POC에서 trigger rate < 50% 면 template-only 전략 재검토 (e.g., 더 공격적 template + LLM 일부 slot만).

### Q2 — Compression Layer B skip 시 extractive 세부

**확정**: **Body 의 `## Intent` 섹션 + `## Behavior` 섹션 각 첫 문단 concat + `tier * multiplier` 토큰 상한 truncate**.

```typescript
// src/recombination/compression/extractive.ts (Phase 2 W3)
export function extractivelySummarize(plasmid: Plasmid, targetTokens: number): string {
  const intent = extractSection(plasmid.body, "Intent")?.split(/\n\n/)[0] ?? "";
  const behavior = extractSection(plasmid.body, "Behavior")?.split(/\n\n/)[0] ?? "";
  const combined = `${intent}\n${behavior}`.trim();
  return truncateToTokens(combined, targetTokens, plasmid.tokenizer);
}
```

**Rationale**:
- Layer A (description 15 tokens) 너무 짧음 — agent 행동 가이드 불충분
- Extractive는 LLM 없이 body 핵심 section 그대로 truncate → 정보 손실 있지만 LLM hallucination 없음
- Section 헤딩 없는 body 에는 첫 200 tokens fallback

**영향**: P-1.13 §3.6 failure mode "로컬 LLM extractive" 행동 구체화 (단순 description fallback 아님).

### Q3 — Cloud `validationParallelism=10` hardcoded

**확정**: **v0.4 default 10. P-1.19 §Q2 `.dhelix/config.json` `execution.validationParallelism` override 제공 — cross-ref 로 해결**.

실제 provider별 rate limit 테이블:
- OpenAI gpt-4o: 500 RPM (실제 10 concurrent OK)
- Anthropic Claude Opus: 50 RPM (10 concurrent 위험 — 권장 5)
- Gemini 1.5: 60 RPM (권장 5)

**Phase 4+ 확장**: `src/llm/providers/rate-limits.ts` 에 provider별 default parallelism 테이블 + auto-detect.

**v0.4 동작**: User가 Anthropic 쓰는데 10 concurrent → rate limit 히트 → auto-backoff. `warnOnce` 로 parallelism 하향 권장.

### Q4 — Local Small `det only` 의 `tool:` expectation

**확정**: **P-1.16 §5.1 cross-ref 와 일관 — `output mentions` 기반 deterministic 재표현**.

LLM auto-gen 이 L1/L2 case 생성 시, `caps.gradingTiers` 가 `["deterministic"]` 만 포함하면 prompt 에 다음 지시:
```
For local-small capability: prefer deterministic-expressible expectations.
- "output contains 'X'" / "output mentions 'X'" (regex based)
- "file:<path> exists" (FS check)
- NOT "tool:<name> is called" (requires tool call trace — semi tier)
```

Eval-seeds 에 이미 `tool:` prefix expectation 있으면 → deterministic grader 시 **"skipped (tool-call trace unavailable)"** 로 표시 + pass 율에서 제외.

**Rationale**: Tool call trace 수집이 agent loop 내부 hook 필요 — local-small 환경 지원 복잡. Skip + 알림이 혼동 없음.
