**이 문서의 역할**: DHELIX.md 재조직 알고리즘의 **marker 규약**, **LLM 프롬프트**, **deterministic fallback**, **I-9 invariance check**를 구현 수준으로 확정.

## 0. TL;DR

- **Marker 규약**: `<!-- BEGIN plasmid-derived: <name> -->` / `<!-- END plasmid-derived: <name> -->`
- Recombination은 marker **쌍 안쪽만** 수정. user-authored (marker 밖) 영역 **절대 불변** (I-9)
- LLM reorg: JSON plan → XML fallback → deterministic fallback (3-tier)
- Invariance check: plan 실행 전 user 영역 hash 비교 → 변경 시 abort
- Cure는 marker 쌍 단위로 제거

## 1. Marker 규약 확정

### 1.1 문법

```markdown
<!-- BEGIN plasmid-derived: <plasmid-name> -->
## <section-heading>

<content>
<!-- END plasmid-derived: <plasmid-name> -->
```

**규칙**:
1. `plasmid-derived:` 접두는 **고정 리터럴** (다른 marker와 충돌 방지)
2. `<plasmid-name>`은 kebab-case, Zod validated
3. BEGIN과 END의 name은 반드시 일치
4. 마커는 **한 줄** (줄 바꿈 금지)
5. BEGIN 직후 반드시 blank line (markdown 파서 호환)
6. 중첩 금지 (한 plasmid 블록 안에 다른 plasmid 블록 불허)

### 1.2 정규식 (parser)

```typescript
const BEGIN_MARKER = /^<!--\s+BEGIN\s+plasmid-derived:\s+([a-z0-9-]+)\s+-->\s*$/gm;
const END_MARKER = /^<!--\s+END\s+plasmid-derived:\s+([a-z0-9-]+)\s+-->\s*$/gm;
```

### 1.3 기타 marker와 충돌 방지

기존 `<!-- markdown-toc -->`, `<!-- prettier-ignore -->` 등과 구분 가능 (접두어 `plasmid-derived:` 유일).

## 2. Section Tree Parser

```typescript
// src/recombination/constitution/section-tree.ts
export interface SectionNode {
  readonly level: number;              // #, ##, ### ...
  readonly title: string;
  readonly content: string;            // heading 제외 본문
  readonly source: "user-authored" | "plasmid-derived";
  readonly plasmidName?: string;       // BEGIN 마커에서 추출 (source = plasmid-derived)
  readonly beginLine?: number;         // BEGIN 마커 line (0-based)
  readonly endLine?: number;
  readonly children: readonly SectionNode[];
  readonly hash: string;               // sha1(title + content), invariance check용
}

export interface SectionTree {
  readonly rootNodes: readonly SectionNode[];
  readonly plasmidDerivedSections: readonly SectionNode[];   // flatten, source="plasmid-derived"
  readonly userAuthoredSections: readonly SectionNode[];      // flatten, source="user-authored"
  readonly rawContent: string;
  readonly contentHash: string;
}

export function parseConstitution(content: string): SectionTree;
```

### 2.1 파싱 알고리즘

1. 라인 단위 스캔. `BEGIN_MARKER` 발견 시 `END_MARKER` (같은 name)까지 block 수집 → `plasmid-derived` 노드
2. marker 밖 `#`, `##`, `###` heading으로 user-authored 섹션 tree 구성
3. heading 없이 시작하는 preamble은 `{level: 0, title: "__preamble__", source: "user-authored"}`
4. Orphan marker (BEGIN만 있고 END 없음) → parse error + 사용자에게 수정 권장

### 2.2 Hash 계산

```typescript
function computeSectionHash(title: string, content: string): string {
  // 공백 정규화 후 hash (whitespace-insensitive)
  const normalized = (title + "\n" + content).replace(/\s+/g, " ").trim();
  return sha1Hex(normalized);
}
```

## 4. LLM-Based Reorg (Tier 1)

### 4.1 프롬프트

```
You are reorganizing DHELIX.md.

Existing DHELIX.md structure (parsed):
${renderSectionTree(input.existingDHELIX)}

Active plasmids (compiled intents):
${renderPlasmidIntentGraph(input.plasmidIntentGraph)}

Previous recombination added these plasmid-derived sections:
${previousDerivedSections.join(", ") || "(none)"}

Rules:
1. PRESERVE user-authored sections exactly (no-marker sections). DO NOT mention or rewrite their content.
2. REMOVE sections in 'previousDerivedSections' that no active plasmid covers anymore.
3. ADD new sections for active plasmids that lack coverage, wrapped in BEGIN/END markers.
4. MERGE semantically overlapping plasmid-derived sections if they come from different plasmids with the same scope.
5. ORDER: user-authored sections keep original position. New plasmid-derived sections insert at end unless plasmid specifies 'location' hint.
6. DO NOT rewrite any existing text — only add/update/remove marker blocks.

Output JSON (no prose):
{ "operations": ReorgOp[], "warnings": string[] }
```

### 4.2 JSON Schema (Zod)

```typescript
const reorgOpSchema = z.discriminatedUnion("op", [
  z.object({ op: z.literal("insert"), after: z.string(), content: z.string(), plasmidName: z.string() }),
  z.object({ op: z.literal("update"), markerName: z.string(), newContent: z.string() }),
  z.object({ op: z.literal("remove"), markerName: z.string() }),
  z.object({ op: z.literal("keep"), sectionTitle: z.string() }),
]);
export const reorgPlanSchema = z.object({
  operations: z.array(reorgOpSchema),
  warnings: z.array(z.string()).default([]),
});
```

### 4.3 Cache Key

```typescript
const cacheKey = sha256([
  input.plasmidIntentGraph.canonicalJSON,
  input.existingDHELIX.contentHash,
  REORG_PROMPT_VERSION,
  input.modelCapabilities.modelDigest ?? "unknown",
].join("\x00"));
```

캐시 위치: `.dhelix/recombination/objects/<hash>.reorg-plan.json`.

## 6. Deterministic Fallback (Tier 3)

LLM 완전 실패 또는 `caps.reliableJson === false` 이면 바로 여기로.

```typescript
// src/recombination/constitution/deterministic-reorg.ts
export function buildDeterministicPlan(input: ReorgInput): ReorgPlan {
  const ops: ReorgOp[] = [];
  const activePlasmids = new Set(input.plasmidIntentGraph.plasmids.map(p => p.name));

  // (1) 이전 derived section 중 현재 active 아닌 것 → remove
  for (const section of input.existingDHELIX.plasmidDerivedSections) {
    if (section.plasmidName && !activePlasmids.has(section.plasmidName)) {
      ops.push({ op: "remove", markerName: section.plasmidName });
    }
  }

  // (2) 활성 plasmid 중 DHELIX.md에 marker 없는 것 → insert at EOF
  for (const plasmid of input.plasmidIntentGraph.plasmids) {
    const hasMarker = input.existingDHELIX.plasmidDerivedSections.some(s => s.plasmidName === plasmid.name);
    if (!hasMarker) {
      ops.push({
        op: "insert",
        after: "__END_OF_FILE__",
        content: renderDefaultSection(plasmid),
        plasmidName: plasmid.name,
      });
    }
  }

  // (3) 기존 derived section 중 plasmid 내용 변경된 것 → update
  for (const section of input.existingDHELIX.plasmidDerivedSections) {
    if (!section.plasmidName) continue;
    const plasmid = input.plasmidIntentGraph.find(section.plasmidName);
    if (!plasmid) continue;
    const expectedContent = renderDefaultSection(plasmid);
    if (section.content.trim() !== expectedContent.trim()) {
      ops.push({ op: "update", markerName: section.plasmidName, newContent: expectedContent });
    }
  }

  return {
    operations: ops,
    warnings: ["LLM reorg failed or unreliable, using deterministic fallback"],
    mode: "deterministic-fallback",
  };
}
```

### 6.1 `renderDefaultSection` 템플릿

```typescript
function renderDefaultSection(plasmid: CompiledPlasmid): string {
  return [
    `## ${defaultHeading(plasmid)}`,
    "",
    plasmid.compressedSummary,   // Layer B 결과
  ].join("\n");
}

function defaultHeading(p: CompiledPlasmid): string {
  if (p.types.includes("structural")) return "Project Constraints";
  if (p.types.includes("value")) return "Core Principles";
  if (p.types.includes("epistemic")) return "Domain Knowledge";
  return titleCase(p.name);
}
```

## 7. I-9 Invariance Check

### 7.1 핵심 규칙

> **User-authored 영역 (marker 밖)이 한 바이트라도 변경되면 plan abort.**

### 7.2 구현

```typescript
// src/recombination/constitution/invariance-check.ts
export function verifyUserAreaInvariance(
  before: SectionTree,
  plan: ReorgPlan,
): { ok: true } | { ok: false; violations: Violation[] } {
  const simulated = simulateApply(before, plan);
  const violations: Violation[] = [];

  // 각 user-authored section의 hash가 before → after에서 변하지 않아야 함
  for (const beforeSection of before.userAuthoredSections) {
    const afterSection = simulated.userAuthoredSections.find(s => s.title === beforeSection.title);
    if (!afterSection) {
      violations.push({ type: "section-removed", title: beforeSection.title });
    } else if (afterSection.hash !== beforeSection.hash) {
      violations.push({ type: "content-changed", title: beforeSection.title, beforeHash: beforeSection.hash, afterHash: afterSection.hash });
    }
  }

  return violations.length === 0 ? { ok: true } : { ok: false, violations };
}
```

### 7.3 Abort 정책

Invariance check 실패 시:
1. Plan 실행하지 않고 Stage 3 preview에서 **ERROR** 표시
2. `REORG_USER_AREA_VIOLATION` error code (PRD §10.3) 발행
3. Transcript에 violation 상세 기록
4. User는 선택:
   - Manual DHELIX.md 편집 후 `/recombination` 재시도
   - `--skip-reorg` 플래그로 reorg stage 우회 (plasmid section bucket은 계속 생성)

### 7.4 Telemetry

```typescript
emitMetric("reorg.user_area_violation_attempt", {
  plasmidName: op.plasmidName,
  violationType: v.type,
  mode: plan.mode,    // "llm-based" vs "deterministic-fallback"
});
```

정상 상태에서 이 metric은 **항상 0**. 1회 이상 발생 → LLM prompt 재검토.

## 8. `simulateApply` — Plan 실행 시뮬레이션

```typescript
// src/recombination/constitution/simulator.ts
export function simulateApply(tree: SectionTree, plan: ReorgPlan): SectionTree {
  let content = tree.rawContent;
  for (const op of plan.operations) {
    switch (op.op) {
      case "insert":   content = applyInsert(content, op); break;
      case "update":   content = applyUpdate(content, op); break;
      case "remove":   content = applyRemove(content, op); break;
      case "keep":     /* no-op */ break;
    }
  }
  return parseConstitution(content);   // re-parse to compute hashes
}
```

**Note**: 실제 file write는 plan validation 통과 후 Stage 4에서만.

## 9. Stage 2d 오케스트레이션 흐름

```
1. parseConstitution(DHELIX.md)           → SectionTree (before)
2. cacheKey = sha256(...)
3. if cache hit: use cached plan
   else:
      (a) if caps.reliableJson:
             plan = tryLLM(JSON)
             if fail: plan = tryLLM(XML)
             if fail: plan = buildDeterministicPlan()
          else:
             plan = buildDeterministicPlan()
      (b) cache.set(cacheKey, plan)
4. verifyUserAreaInvariance(before, plan)
      if violations: abort + REORG_USER_AREA_VIOLATION error
5. emit telemetry (reorg.mode, cache_hit, etc.)
6. return plan → Stage 3 Preview
```

## 10. Edge Cases

| 상황 | 대응 |
|-----|-----|
| DHELIX.md 없음 | empty SectionTree → 모든 plasmid가 신규 insert |
| DHELIX.md가 marker만 있고 user 영역 0 | OK. `userAuthoredSections` 배열 [] |
| 같은 plasmid에 BEGIN/END 두 쌍 (중복) | parse error |
| BEGIN만 있고 END 없음 (orphan) | parse error — `DHELIX_MD_MALFORMED` |
| LLM이 `op: "insert"` with `after: "존재하지 않는 섹션"` | simulateApply 실패 → deterministic fallback |
| LLM이 user-authored 제목을 `update` 대상으로 지정 | verifyUserAreaInvariance 잡힘 → abort |
| `plasmidIntentGraph` 비어있음 (active plasmid 0) | 기존 derived section 전부 remove, user 영역만 남음 |

## 11. 테스트 계획

### 11.1 Unit

- `parseConstitution.spec.ts`:
  - Empty file → empty tree
  - 1 user section + 1 plasmid-derived → 올바른 분류
  - Orphan BEGIN → parse error
  - Nested markers → parse error
- `buildDeterministicPlan.spec.ts`:
  - Active plasmid 3개, 기존 derived 2개 (1개 orphan) → 1 remove + 1 insert + 1 update
- `verifyUserAreaInvariance.spec.ts`:
  - User section unchanged → ok
  - Plan이 user section hash 변경 → violation
  - User section removed → violation

### 11.2 Integration

- LLM (stub): plan JSON 반환 → apply → 결과 확인
- LLM fails JSON → XML fallback → plan 성공
- LLM returns violating plan (user area 수정 시도) → abort

## 12. Resolved Questions (v0.2)

### Q1 — `plasmid.location` hint 채택

**결정**: **단순 `after: <heading-title>` 형식으로 v0.4 포함**.

**Schema 확장 (`src/plasmids/schema/v1.ts`)**:
```yaml
location:
  after: "## Architecture"    # optional. 이 heading 다음에 삽입.
```
```typescript
location: z.object({
  after: z.string().regex(/^#{1,6}\s+.+/),   // "## Title" 형식 강제
}).optional()
```

**LLM 프롬프트 rule 추가 (§4.1 rule 7 신설)**:
```
7. USE 'location.after' hint if provided and matches an existing heading in DHELIX.md.
   If hint is given but no heading matches, fall back to EOF and include in plan.warnings.
```

**Deterministic fallback (§6 `buildDeterministicPlan`)**:
```typescript
const after = plasmid.location?.after
  && input.existingDHELIX.userAuthoredSections.some(s => heading(s) === plasmid.location.after)
    ? plasmid.location.after
    : "__END_OF_FILE__";
ops.push({ op: "insert", after, content: ..., plasmidName: plasmid.name });
```

**Rationale**:
- EOF append only → 새 plasmid-derived section이 파일 끝으로 몰림 → DHELIX.md 가독성 저하
- Before/between/relative 복잡 DSL은 over-engineering. 단일 anchor로 90% 케이스 해결
- 미매칭 시 **fail-fast 아님** → EOF fallback + warning. 사용자 typo 방어

**영향**:
- `src/recombination/constitution/reorganizer.ts` `ReorgOp.insert.after` 값 선택 로직 확장
- Warning metric: `reorg.location_hint_miss_total{plasmid}`
- LOC: +10 (LLM prompt rule + fallback 분기)

### Q2 — `update` op의 user heading marker 전환 방어

**결정**: **거부 유지 + plan parsing 단계 조기 거부 검증 추가** (`validateUpdateTargets`).

**신규 함수** (`src/recombination/constitution/plan-validator.ts`):
```typescript
export function validateUpdateTargets(tree: SectionTree, plan: ReorgPlan): void {
  const knownMarkers = new Set(
    tree.plasmidDerivedSections.map(s => s.plasmidName!).filter(Boolean),
  );
  for (const op of plan.operations) {
    if (op.op !== "update") continue;
    if (!knownMarkers.has(op.markerName)) {
      throw new ReorgPlanError(
        `REORG_INVALID_UPDATE_TARGET: update op targets "${op.markerName}" ` +
        `which is not an existing plasmid-derived marker. ` +
        `Possible I-9 violation attempt.`,
      );
    }
  }
}
```

**호출 순서 (§9 Stage 2d 흐름 수정)**:
```
3. LLM/deterministic plan 생성
3.5. validateUpdateTargets(before, plan)     ← v0.2 신규 — 조기 거부
4. verifyUserAreaInvariance(before, plan)    ← 최후 방어선 (기존)
```

**Rationale**:
- 최후 방어선 `verifyUserAreaInvariance` 는 semantic hash 비교 — 비용 높음
- Plan parsing 직후 구조적 검증 (markerName 존재?) 으로 **98%의 위반 시도 조기 차단**
- LLM이 user heading title을 markerName으로 사용하려는 패턴 (가장 흔한 I-9 위반 방식) 잡힘
- 이중 방어로 robustness 확보

**영향**:
- PRD §10.3 Error Code Catalog에 `REORG_INVALID_UPDATE_TARGET` 추가
- `src/recombination/constitution/plan-validator.ts` 신규 (~30 LOC)
- Unit test: "LLM이 'My Architecture' markerName update 시도 → 조기 거부" 케이스

### Q3 — Top-level `# DHELIX.md` heading 자연 보호

**확정**: **Parser 규칙으로 자연 보호 — 특수 처리 불필요**.

**SectionTree parser 불변식**:
1. `BEGIN_MARKER` 는 `##` 이상 heading **앞에만** 배치 가능 (`#` heading 앞 배치 시 parse error)
2. 문서 시작의 `# <title>` 은 항상 level 1 user-authored section (첫 번째 `userAuthoredSections` 엔트리)
3. 모든 `plasmidDerivedSections` 는 level ≥2

**Parser validation**:
```typescript
function parseConstitution(content: string): SectionTree {
  // ...
  for (const marker of beginMarkers) {
    const nextHeadingLine = findNextHeading(content, marker.line);
    const level = countHashes(nextHeadingLine);
    if (level === 1) {
      throw new ParseError(`BEGIN_MARKER before level-1 heading not allowed (line ${marker.line})`);
    }
  }
  // ...
}
```

**Rationale**:
- 명시적 top-level 보호 로직은 불필요 — marker convention 이 **level ≥2 에만 적용**되므로 `#` 영역 자동 보호
- 사용자가 여러 `# Section` (level 1) 작성해도 모두 user-authored로 분류
- Constraint: plasmid-derived content 는 document-level (`#`) 직접 만들 수 없음 — 시맨틱상 올바름 (plasmid는 subsection만 제안)

**영향**:
- `src/recombination/constitution/section-tree.ts` parse 시 constraint 명시 (§2.1)
- Parser unit test: `"# Root\n<!-- BEGIN plasmid-derived: foo -->\n# Sub"` → parse error
- Unit test: `"# Root\n## Section"` → rootSections[0].level=1, userAuthoredSections 2개 (둘 다 user)

### Q4 — MERGE semantic 제거 — v0.4 순차 stack 채택

**결정**: **v0.4에서 MERGE 규칙 제거**. 같은 bucket 내 다중 plasmid는 순차 BEGIN/END stack 유지.

**LLM 프롬프트 rule 4 재작성 (§4.1)**:
```
[기존 v0.1] 4. MERGE semantically overlapping plasmid-derived sections if they come from different plasmids with the same scope.

[v0.2] 4. KEEP each plasmid-derived section as a separate BEGIN/END block.
         Do NOT merge content from different plasmids, even if they appear redundant.
         Cure must be able to remove individual plasmid blocks without affecting others.
```

**구체 이유**:
- "의미 겹침" 판정은 본질적 LLM 판단 영역 → 결정성 상실 위험
- Merge 된 marker block의 cure 시 **부분 삭제 복잡** — 어느 plasmid를 cure하면 block의 어떤 부분만 제거?
- 현재 설계 (§4.2 순차 stack) 는 단순 + cure 안전성 보장 + 사용자에게 투명 (어느 plasmid가 어느 section 유발)

**Deferred to Phase 4+ (GAL-3 Chemistry Laws)**:
- PRD §26 Plasmid Chemistry 구현 시점에 **명시적 merge 규칙** 설계
- `compatible-with` / `conflicts-with` 필드 기반 explicit merge 만 허용 (LLM 판단 아님)
- Object store의 content-addressed merge 결과를 별도 plasmid-group으로 취급

**영향 (v0.4)**:
- LLM 프롬프트 `rule 4` 교체 (§4.1)
- Test case 추가: "2 plasmid 같은 bucket, 다른 description → 2 BEGIN/END block 각각 생성 (merge 안 함)"
- Cure 단순 (PRD §6.4): marker name 기반 단일 block 제거
