## 0. TL;DR

- Advisory file lock — `O_EXCL | O_CREAT` atomic create
- TTL 10분 (600s). 경과 시 stale → 자동 cleanup
- Lock 정보: pid, hostname, command, acquiredAt, expiresAt
- Crash recovery: `*.partial` / `*.tmp` 감지 → 사용자 안내 + rollback 제안
- 동시 실행 시 두 번째 프로세스는 `LockHeldError` 에러 (blocking 아님, fast-fail)

## 1. Lock File Path & Schema

```typescript
// src/recombination/lock.ts
const LOCK_FILE = ".dhelix/recombination/.lock";

export const lockInfoSchema = z.object({
  pid: z.number().int(),
  hostname: z.string(),
  command: z.enum(["recombination", "cure", "plasmid-write"]),
  acquiredAt: z.string().datetime(),
  expiresAt: z.string().datetime(),     // acquiredAt + 10min
  version: z.literal(1),
});
export type LockInfo = z.infer<typeof lockInfoSchema>;
```

## 2. Acquire 알고리즘

```typescript
export async function acquireLock(command: LockCommand): Promise<LockHandle> {
  const info: LockInfo = {
    pid: process.pid,
    hostname: os.hostname(),
    command,
    acquiredAt: new Date().toISOString(),
    expiresAt: new Date(Date.now() + LOCK_TTL_MS).toISOString(),
    version: 1,
  };

  try {
    // O_EXCL | O_CREAT (node: flag "wx") — atomic, fails if file exists
    await writeFile(LOCK_FILE, JSON.stringify(info, null, 2), { flag: "wx" });
  } catch (err: any) {
    if (err.code === "EEXIST") {
      // Existing lock — check staleness
      const existing = await readLockFile();
      if (new Date(existing.expiresAt) < new Date()) {
        await cleanupStaleLock(existing);
        return acquireLock(command);   // one retry
      }
      throw new LockHeldError(existing);
    }
    throw err;
  }

  return new LockHandle(LOCK_FILE, info);
}

async function cleanupStaleLock(existing: LockInfo): Promise<void> {
  logWarn(`Stale lock cleaned up: pid=${existing.pid} hostname=${existing.hostname} expired at ${existing.expiresAt}`);
  await unlink(LOCK_FILE);
}
```

**Lock TTL 10분 근거**:
- Cloud recombination ~2분
- Local recombination ~10분 (P-1.20 §3)
- 10분은 "최악 시나리오 + 버퍼"

TTL 연장 필요 시 `/recombination --lock-ttl 30m` 옵션 (Phase 4+).

## 3. Release

```typescript
export class LockHandle {
  constructor(private readonly path: string, private readonly info: LockInfo) {}

  async release(): Promise<void> {
    try {
      const current = JSON.parse(await readFile(this.path, "utf-8")) as LockInfo;
      if (current.pid !== this.info.pid) {
        // 다른 프로세스가 이미 점유 → 우리 lock 아님, release하지 않음
        logWarn(`Lock was reacquired by another process (pid ${current.pid}), skipping release`);
        return;
      }
      await unlink(this.path);
    } catch {
      // 이미 삭제됨 등 OK
    }
  }
}
```

**주의**: `try/finally`로 항상 release 호출 필요:
```typescript
const lock = await acquireLock("recombination");
try {
  await runRecombination();
} finally {
  await lock.release();
}
```

## 4. Stale Lock 판정 시나리오

| 상황 | `acquiredAt` 대비 현재 시간 | 판정 |
|-----|------------------------|-----|
| 3분 전 획득, 같은 머신, 다른 pid | 3min < 10min | active — LockHeldError |
| 3분 전 획득, 같은 pid (자기 자신) | 동일 | 재사용 허용 (중첩 acquire는 no-op) |
| 15분 전 획득 | 15min > 10min | stale — cleanup + retry |
| 다른 hostname (CI 머신 vs 로컬) | TTL 경과 전이면 | LockHeldError (명시 "running on CI machine...") |

### 4.1 진단 가능한 Lock 메시지

```typescript
export class LockHeldError extends Error {
  constructor(public readonly holder: LockInfo) {
    super(
      `Recombination lock held by another process:\n` +
      `  pid: ${holder.pid}\n` +
      `  hostname: ${holder.hostname}\n` +
      `  command: ${holder.command}\n` +
      `  acquired: ${holder.acquiredAt}\n` +
      `  expires: ${holder.expiresAt}\n` +
      `\n` +
      `Options:\n` +
      `  1. Wait until ${holder.expiresAt}\n` +
      `  2. If process is dead, delete ${LOCK_FILE} manually\n` +
      `  3. Use --force-unlock (not recommended)`
    );
  }
}
```

## 7. VS Code Extension + CLI 공존

CI와 로컬, VS Code 확장과 CLI가 같은 프로젝트에서 동시에 작업 — 시나리오 커버:

| 시나리오 | 대응 |
|---------|-----|
| 로컬 CLI가 lock held, CI가 시도 | CI의 `acquireLock` → LockHeldError → fail-fast (CI job exit 1) |
| VS Code 확장이 file watch, CLI가 recombination 실행 중 | 확장은 lock 존재 확인 후 "in progress" UI 표시, 자체 작업은 대기 |

## 9. Resolved Questions (v0.2)

### Q1 — Same-pid 중첩 acquire

**결정**: **허용 (reentrant)**. 같은 pid + 같은 command 재호출 시 no-op + log. 다른 command (예: recombination 중 cure) 는 `LockReentryError` throw.

**Rationale**: `/plasmid` 내부 `/recombination` 호출은 자연스럽지만 cure 중 recombination은 트랜잭션 무결성 훼손. 명시 분기.

### Q2 — TTL 10분 적정성 (local-small)

**결정**: **15분으로 상향 + `--lock-ttl` CLI 플래그**. Local-small 최악 시나리오 (10 plasmid × 1 min each)에서도 여유. Power user는 `--lock-ttl 30m` override 가능.

**Rationale**: Phase 0 POC Ollama 실측 전 선보수적 상향. 초과 시 graceful retry 없이 fail-fast 유지 (debug 용이).

**Constant 분리**: `src/recombination/lock.ts` `export const LOCK_TTL_MS = 15 * 60_000` (POC 후 조정 가능).

### Q3 — Git worktree 격리

**결정**: **v0.4 공유 `.dhelix/` 유지 + 경고 감지**. 여러 worktree에서 `.dhelix` 경로가 realpath 동일하면 "worktree lock shared — consider per-worktree `.dhelix/`" warning.

**Rationale**: Worktree별 `.dhelix` 복제는 plasmid 동기화 부담. v0.4는 단순성 우선, v0.5에서 `dhelix.worktreeMode: "shared" | "isolated"` config 옵션 도입.
