## P-1.9 — "Intent-authored" 슬로건 (PRD §31.2)

### 기대 (Hardening v1 원 요구)

v0.2 슬로건 "authored, not configured"의 문제:
- AI 엔진이 끼어드는 과정이 가려짐 → 사용자가 모든 것을 손수 짠 것처럼 오해
- "Configured vs Authored" 이분법이 과장됨

**v0.3 목표**: 슬로건이 "의도 저작 + 컴파일" 두 측면을 정직하게 담아야 한다.

### PRD §31.2 현 상태 (검증)

```
> dhelix-code is the first coding agent that treats user intent as a first-class,
> compilable artifact — verified on each build, reversible on demand,
> hermetically separated from runtime.

"First-class artifact" — configuration(문자열 덩어리)이 아니라 컴파일 가능한 코드 같은 존재.

Footnote: "Authored" refers to intent. Compilation from intent to artifact is
assisted by dhelix's interpreter. The compiler can run locally (Ollama) or in
the cloud, at your choice.
```

### 검증 결과: ✅ 반영 완료

- "first-class, **compilable** artifact" — 컴파일 측면 명시
- "**verified** on each build" — Stage 6 validation 함의
- "**reversible** on demand" — Cure 함의
- "**hermetically** separated from runtime" — I-8 Hermeticity 직접 언급
- Footnote가 LLM interpreter의 존재를 명시 → 정직함 유지

**남은 검토 포인트**: Landing page / README / release notes 등 **PRD 외 공개 자료**에서도 일관되게 이 슬로건 사용. Phase 6 (Polish & Dogfood)에서 점검.

## P-1.11 — §21.2 Biology Table 3-Column (PRD §21.2)

### 기대 (Hardening v1 원 요구)

v0.2의 "Biology = Plasmid System" 등치가 과장됨. **Descriptive only** 원칙 명시 + 표 형식으로 "Biology concept / Plasmid system 적용 / Status" 3-column 구조.

### PRD §21.2 현 상태 (검증)

PRD line 1803~1820 확인:
- Table 3-column: Biology 개념 / Plasmid 적용 / Status (descriptive/enforced/metaphor only)
- 각 row에 "descriptive only"인지 "engineering enforced"인지 명시
- §20.2의 "원칙"에서도 "Biology metaphor는 descriptive only — 엔지니어링 정당화를 먼저"

### 검증 결과: ✅ 반영 완료

- 3-column 구조 적용됨
- "Nuclear membrane" → "I-8 Hermeticity (기술적 강제)" — enforced 명시
- 대부분의 biology 개념은 "설계 규범" 또는 "서술적 유비" 로 분류
- 엔지니어링 enforcement와 메타포가 명확히 구분됨

**남은 검토 포인트**: Part II §21 이후 철학/메타포 섹션에서 biology 용어 사용 시 "이건 비유" 주석 일관성. Phase -1 Week 3-4 carryover 검토에서 spot-check (P-1.12와 함께).

## P-1.12 — Rhizome / Friction Tier 사상 정합 (PRD §20.2, §27.2)

### 기대 (Hardening v1 원 요구)

v0.2의 이슈: Deleuze "Rhizome (수평)" 개념과 "Tier 체계 (foundational / policy / tactical, 위계적)" 가 모순되어 보임.

**v0.3 목표**: 두 개념이 모순 없이 공존한다고 정당화. 구체적으로:
- Rhizome은 **plasmid 간 관계** (composition, extends, conflicts-with)에만 적용
- Tier는 **"변경 마찰" 의 차등**이지 권력 위계가 아님

### PRD §20.2 현 상태 (검증)

line 1710~1731:
- Rhizome row: "Plasmid 간 수평 관계 (composition, chemistry)" — 수평성을 **plasmid 관계**로 한정
- 원칙 3개 명시:
  1. 사상들은 **선택적 차용** (부분 적용)
  2. "Rhizome의 수평성은 plasmid 간 관계에만 적용, tier는 '변경 마찰'의 차등이지 권력 위계 아님"
  3. Biology metaphor는 descriptive only

### PRD §27.2 현 상태 (검증)

line 2137~2145:
```
### 27.2 Friction Tier Model (v0.3 재해석)

| Tier | 의미 (변경 마찰) |
|-----|---------------|
| Tier 1: Mandatory | 팀 requiredPlasmids — 개인 비활성화 불가 |
| Tier 2: Recommended | 기본 활성, 끌 수 있되 사유 기록 |
| Tier 3: Optional | 활성화 자유 |

중요: Tier는 권력 위계가 아니라 변경 마찰. Rhizome과 양립 (§20.2).
```

### 검증 결과: ✅ 반영 완료

- §20.2가 Rhizome의 수평성 범위를 **plasmid 간 관계**로 한정
- §27.2가 Tier를 "변경 마찰 차등"으로 명확히 재정의
- 두 섹션 간 cross-reference ("Rhizome과 양립 §20.2") 성립

**남은 검토 포인트**: `tier: foundational` 과 team's `Tier 1: Mandatory` 가 서로 다른 개념임을 문서에서 구분 명확. Phase 5 (Foundational + `/plasmid challenge`) 구현 시 터미널 UX에서도 "friction tier" vs "team tier" 혼동 없게 네이밍 점검.
