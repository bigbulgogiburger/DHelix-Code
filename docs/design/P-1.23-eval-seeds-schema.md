## 2. PlasmidEvalSeedSchema

```typescript
// src/plasmids/schema/eval-seed.ts (신규)
import { z } from "zod";
import { evalCaseSchema } from "../../skills/creator/evals/types.js";

export const evalTierSchema = z.enum(["L1", "L2", "L3", "L4"]);
export type EvalTier = z.infer<typeof evalTierSchema>;

export const plasmidEvalSeedSchema = evalCaseSchema.extend({
  /** Validation tier (L1-L4). Phase 1+의 volume governor가 이 값 기반 분류. */
  tier: evalTierSchema,
});

export type PlasmidEvalSeed = z.infer<typeof plasmidEvalSeedSchema>;

/** Plasmid frontmatter의 eval-seeds 배열 스키마 */
export const plasmidEvalSeedsArraySchema = z.array(plasmidEvalSeedSchema).optional();
```

**호환성 보장**:
- 기존 `evalCase`의 모든 필드 유지 (id, prompt, files, expectations, ...)
- `tier` 필드만 추가 (enum, strict)
- `trigger_only`, `should_trigger` 기본값 유지 — plasmid seeds도 동일 의미로 활용 가능

## 4. Expectation DSL (검증 문법)

`expectations` 배열의 각 문자열은 **자유 자연어 + 선택적 prefix**:

| Prefix | 의미 | 채점 방식 |
|-------|-----|---------|
| `tool:<name>` | 해당 tool이 호출됨 | gradeByToolCallTrace (deterministic semi) |
| `hook:<event>` | 해당 hook이 발동됨 | gradeByToolCallTrace |
| `output contains "<str>"` / `output mentions <kw>` | regex / substring match | gradeByRegexPattern (deterministic) |
| `output does NOT contain "<str>"` | negation | 동일 |
| `file:<path> exists` / `file:<path> modified` | file system effect | gradeByFileSystemEffect |
| `exit code <n>` | process exit code | gradeByExitCode |
| (prefix 없음) | 자연어 — LLM judge fallback | gradeByLlmJudge (local 시 skip) |

**설계 원칙**: prefix가 명확할수록 deterministic grader가 처리 → LLM judge 의존 최소. 로컬 LLM 환경 안전성 향상 (PRD §8.4).

Parser는 Phase 2에 `src/recombination/validation/expectation-parser.ts`에서 구현. P-1.16 Validation framework에서 상세.

## 5. Frontmatter Parser 통합

### 5.1 기존 구조

`src/plasmids/schema/v1.ts` (Phase 1 W1 신규) 에 core 6 + extensions:

```typescript
export const plasmidCoreSchema = z.object({
  name: z.string().regex(/^[a-z0-9-]+$/),
  description: z.string().min(10).max(500),
  active: z.boolean().default(true),
  scope: z.array(z.enum(["hooks","rules","skills","commands","agents","harness"])).min(1),
  priority: z.enum(["low","normal","high"]).default("normal"),
  types: z.array(z.enum(["behavioral","structural","ritualistic","epistemic","value"])).min(1),
});

export const plasmidExtensionsSchema = z.object({
  tier: z.enum(["foundational","policy","tactical","agent-proposed"]).default("policy"),
  // ... compatible-with, conflicts-with, extends, ...
  "eval-seeds": plasmidEvalSeedsArraySchema,
  // ... source, privacy, prompt-compression, artifacts, etc.
}).passthrough();  // 알려지지 않은 필드는 ignore (forward compat)
```

### 5.2 Parse 흐름

```typescript
// src/plasmids/frontmatter.ts (Phase 1 신규)
import { parse as yamlParse } from "yaml";

export interface ParsedPlasmid {
  readonly core: PlasmidCore;
  readonly extensions: PlasmidExtensions;
  readonly body: string;
}

export function parsePlasmidFrontmatter(raw: string): ParsedPlasmid {
  const { frontmatter, body } = extractFrontmatter(raw);
  const yaml = yamlParse(frontmatter);

  const core = plasmidCoreSchema.parse(yaml);
  const extensions = plasmidExtensionsSchema.parse(yaml);

  return { core, extensions, body };
}
```

**에러**: Zod validation 실패 시 `PLASMID_SCHEMA_VIOLATION` error code (PRD §10.3) + 각 필드별 상세 메시지.

### 5.3 eval-seeds 전용 에러 케이스

| 실패 | Zod 메시지 예시 | 사용자 수정 힌트 |
|-----|----------------|----------------|
| `tier`가 enum 밖 (예: `L5`) | `"Invalid enum value. Expected 'L1' \| 'L2' \| 'L3' \| 'L4', received 'L5'"` | "L1~L4만 지원. L5는 예약됨." |
| `expectations` 빈 배열 | `"Array must contain at least 1 element(s)"` | "각 seed에 1개 이상 expectation 필수" |
| `id` 중복 (여러 seed에 같은 id) | custom refine 추가 필요 (아래 §6) | "seed id는 plasmid 안에서 유일" |
| `prompt` 빈 문자열 | `"String must contain at least 1 character(s)"` | "prompt 필수" |

## 6. Seed ID 중복 검사 (custom refine)

```typescript
export const plasmidEvalSeedsArraySchema = z.array(plasmidEvalSeedSchema).optional()
  .superRefine((seeds, ctx) => {
    if (!seeds) return;
    const seenIds = new Set<string>();
    for (const seed of seeds) {
      if (seenIds.has(seed.id)) {
        ctx.addIssue({
          code: z.ZodIssueCode.custom,
          message: `Duplicate eval-seed id: "${seed.id}"`,
          path: ["eval-seeds"],
        });
      }
      seenIds.add(seed.id);
    }
  });
```

## 11. Resolved Questions (v0.2)

### Q1 — `tier` 필드 required vs default

**확정**: **Required (강제). 생략 시 Zod validation error + "tier 명시 권장" 안내**.

```typescript
// src/plasmids/schema/eval-seed.ts
export const plasmidEvalSeedSchema = evalCaseSchema.extend({
  tier: evalTierSchema,    // required, no default
});
```

Error message 커스터마이징:
```typescript
evalTierSchema.describe(
  "L1 (direct) | L2 (indirect) | L3 (contextual) | L4 (adversarial)"
)
```

**Rationale**:
- Default L1 은 "안전한 기본값" 같지만 실제로는 L2/L3/L4 작성 기회를 놓침
- Validation 품질은 tier 분포 다양성에 비례 → 사용자가 의식적으로 tier 고르는 교육 효과
- Required = schema 명확성 + 후속 volume governor 입력 안정

### Q2 — L3 seed 자동 생성

**확정**: **P-1.16 §2 case-generator 에서 LLM auto-gen 지원. 사용자 수동 eval-seeds 우선 소비 후 remaining quota 자동 채움**.

생성 priority (P-1.16 §2.1 재확인):
1. `plasmid.eval-seeds` 의 L3 tier 엔트리 (사용자 수동)
2. `plasmid.expression-conditions` 필드 기반 deterministic 추출 (branch test)
3. LLM auto-gen — `## Constraints` 섹션에서 "exception/조건부" 문구 추출 기반

**Rationale**: L3 는 "조건부 로직 정확도" — 사용자가 구체 시나리오 가장 잘 앎. Auto-gen 은 보완.

**Cross-ref**: P-1.16 §2.2 Deterministic extraction 규칙 (L3 부분).

### Q3 — `expected_output_contains` / `_excludes` vs expectation DSL 중복

**확정**: **양쪽 모두 유지. skill-creator 호환성 유지하되 plasmid 에서는 expectation DSL 권장**.

```typescript
// Backwards compat — legacy fields
expected_output_contains?: string[];       // array of substrings → "output contains 'X'" DSL과 동등
expected_output_excludes?: string[];       // → "output does NOT contain 'X'"

// Plasmid 전용 DSL 권장
expectations: ["output contains 'X'", "tool:/security-scan is called"];
```

Grader 구현: legacy 필드를 expectation DSL 로 자동 변환 후 cascade 적용.

**Rationale**:
- Legacy 필드 제거 시 skill-creator evals 의 기존 테스트 파괴
- Plasmid 전용 필드는 expectation DSL (더 표현력 있음) 권장
- 두 방식 공존 → grader 단순화 (모두 DSL 로 정규화 후 처리)

### Q4 — eval-seeds 최대 개수 상한

**확정**: **plasmid 당 20 seeds (Zod `max(20)`)**. 초과 시 schema violation.

```typescript
export const plasmidEvalSeedsArraySchema = z.array(plasmidEvalSeedSchema)
  .max(20, { message: "eval-seeds limit is 20 per plasmid. Use volume governor for additional cases." })
  .optional()
  .superRefine((seeds, ctx) => { /* duplicate id 검사 */ });
```

**Rationale**:
- 20 seeds × 10 plasmid = 200 seeds → volume governor 150 case 상한과 비례
- 사용자가 수백 seeds 작성하려는 패턴은 validation framework 오용 (별도 test suite 가 적절)
- 20 초과 필요하면 volume governor가 LLM auto-gen 으로 보강 가능
