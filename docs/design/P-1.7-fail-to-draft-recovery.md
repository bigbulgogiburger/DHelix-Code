## 0. TL;DR

- 7 주요 error code → recovery strategy 매핑 (§1)
- 각 에러에서 **사용자가 다음에 할 수 있는 액션 2-4개** 제시
- Common pattern: "원인 설명 → 2-4 option → 사용자 선택 → action"
- "Template fallback" — LLM 실패 시 빈 템플릿에 FM만 채워서 edit 유도

## 1. Error Code → Recovery 매핑

| Error code (PRD §10.3) | Stage | 주요 원인 | Recovery options |
|----------------------|-------|---------|-----------------|
| `PLASMID_PARSE_ERROR` | /plasmid quick | LLM 출력 형식 불량, frontmatter 깨짐 | (a) minimal template + editor (b) retry with simpler intent (c) research mode (d) abort |
| `PLASMID_SCHEMA_VIOLATION` | /plasmid validate, /recomb stage 1 | Zod field 불일치 | (a) 필드별 오류 highlight + inline fix suggestion (b) editor open at error position (c) auto-fix (only safe defaults) (d) abort |
| `RECOMBINATION_PLAN_ERROR` | stage 2a interpret | Interpreter LLM 실패 | (a) retry with chunked strategy (b) retry with more retries (c) switch to cloud (if cascade enabled) (d) skip this plasmid |
| `INTERPRETER_JSON_FAILURE` | stage 2a | JSON parse 반복 실패 | (a) XML fallback (auto) (b) field-by-field split (c) switch model (d) manual plan.json edit |
| `GENERATOR_ERROR` | stage 2b | Template slot fail | (a) template defaults (b) template-only mode (c) skip this artifact (keep plasmid) (d) abort |
| `REORG_USER_AREA_VIOLATION` | stage 2d | LLM이 user 영역 수정 시도 | (a) deterministic fallback (auto) (b) skip reorg stage (c) manual DHELIX.md edit (d) abort |
| `VALIDATION_FAILED_L1` / `L2` | stage 6 | Validation threshold 미만 | (a) rollback (default, 10s grace) (b) inspect failed cases (c) re-run with cloud (d) edit plasmid to simplify |

```

  Cause: LLM produced malformed frontmatter
  Last attempt:
    ---
    name: owasp-gate
    description: [unfinished ...
    ---

Recovery options:
  [1] Use minimal template — we'll prefill 6 required fields, you edit the body
  [2] Retry with simpler intent (shorter, 1 sentence)
  [3] Switch to --research mode (structured questionnaire, 3-5 min)
  [4] Abort

> 1

Creating .dhelix/plasmids/owasp-gate.md with minimal template:
  ---
  name: owasp-gate
  description: "OWASP Top 10 강제"
  scope: [hooks, rules]
  types: [structural]
  priority: normal
  tier: policy
  active: true
  ---

  ## Intent
  (편집해주세요)

  ## Behavior
  (편집해주세요)

Opening $EDITOR...
```

### 2.6 REORG_USER_AREA_VIOLATION (I-9 위반 감지)

```
✗ Stage 2d: Constitution Reorganizer produced a plan that modifies user-authored sections
  Violation:
    - section "## My Architecture" (user-authored)
      Before hash: abc123  After hash: def456

Auto-fallback:
  [auto] Using deterministic reorg plan instead (LLM plan discarded)
  [auto] Telemetry: reorg.user_area_violation_attempt +1

Recovery options:
  [1] Continue with deterministic plan (recommended, safe)
  [2] Skip reorg stage entirely (prompt sections will be generated but DHELIX.md unchanged)
  [3] Abort and inspect .dhelix/recombination/.drafts/reorg-plan.json
  [4] Manually edit DHELIX.md then re-run

> 1
  ✓ Deterministic plan applied. Stage 3 Preview starting...
```

### 2.7 VALIDATION_FAILED_L1 / L2 (Runtime validation)

```
✗ Stage 6: Validation failed — L1 17/20 (85%) below 90% threshold

Pattern analysis:
  - 3 failed cases all use pattern: "output mentions security"
  - Grader type: deterministic (regex) — not an LLM uncertainty
  - Plasmid 'no-verbose-logging' is causing agent to skip security mention

Hypothesis: plasmid 'no-verbose-logging' may conflict with 'owasp-gate'.

Recovery options (10s grace period):
  [r] Rollback (default, auto in 10s) — revert all artifacts
  [k] Keep despite failure (+audit log override reason required)
  [c] Re-run with cloud model (may produce different results)
  [i] Inspect failed cases
  [e] Edit plasmid 'no-verbose-logging' to exclude security logs
  [s] Suggest plasmid edit (we analyze + propose diff)

> s

Proposed edit for .dhelix/plasmids/no-verbose-logging.md:
  ## Behavior
- Suppress all verbose log output during code review.
+ Suppress verbose log output during code review, EXCEPT security-related
+ messages (OWASP findings, auth issues) which must remain visible.

Apply this edit? [y/n/e (manual edit)]
```

## 3. Recovery Strategy API

```typescript
// src/recombination/recovery/strategy.ts
export interface RecoveryContext {
  readonly errorCode: ErrorCode;
  readonly stage: Stage;
  readonly plasmidName?: string;
  readonly error: unknown;
  readonly capabilities: ModelCapabilities;
  readonly strategies: PipelineStrategies;
  readonly transcriptId: string;
}

export interface RecoveryOption {
  readonly key: string;              // "1" | "2" | "r" | "k" | ...
  readonly label: string;
  readonly description: string;
  readonly action: () => Promise<RecoveryResult>;
  readonly isDefault?: boolean;       // grace period default
  readonly warnings?: readonly string[];   // "sends to cloud", "permanent action"
}

export interface RecoveryResult {
  readonly outcome: "recovered" | "skipped" | "aborted";
  readonly continuePipeline: boolean;
  readonly artifactMutation?: "none" | "generated" | "rolled-back";
}

export function getRecoveryOptions(ctx: RecoveryContext): readonly RecoveryOption[];
export async function promptAndExecute(ctx: RecoveryContext): Promise<RecoveryResult>;
```

## 4. Auto-Fix Heuristics (옵션 "[1] Auto-fix" 에 사용)

### 4.1 Schema violation auto-fixes

| Field | 위반 | Auto-fix |
|------|-----|---------|
| `scope` | 단수형 (예: "hook") | 복수형 ("hooks") |
| `scope` | 알려진 typo (예: "agnt") | closest match ("agents") via Levenshtein distance ≤ 2 |
| `types` | 빈 배열 | description 기반 inference (P-1.5 §2.1) |
| `types` | 단일 string 대신 배열? | single-element array wrap |
| `description` | 짧음 (<10 chars) | 사용자 prompt 재실행, no auto-fix |
| `description` | 길음 (>500 chars) | 첫 1-2 문장만 취함 |
| `priority` | 알려진 변형 ("Normal", "NORMAL") | lowercase |
| `tier` | 미설정 + `challengeable-by` 있음 | `tier: foundational` 유추 |
| `name` | 파일명과 불일치 | 파일명 기준 수정 |
| `name` | CamelCase | kebab-case 변환 |

### 4.2 안전하지 않은 auto-fix는 제안만

| 상황 | 자동 적용 안 함 |
|-----|--------------|
| `types: []` 라서 inference 필요 | 제안만 (yes/no 확인) |
| `tier: foundational` 누락 인퍼런스 | 제안만 |
| 미지 필드 (`passthrough`로 제거 불가) | 경고 후 user에게 |

## 6. Suggest Plasmid Edit (`[s]`)

가장 고급 기능. Validation 실패 패턴을 LLM이 분석하여 plasmid edit diff 제안.

```typescript
// src/recombination/recovery/suggest-edit.ts
export async function suggestPlasmidEdit(
  failedCases: readonly GradedCase[],
  plasmid: Plasmid,
  llm: LLMClient,
): Promise<{ diff: string; rationale: string } | null>;
```

**프롬프트**:
```
You analyzed validation failures for plasmid "${plasmid.name}".

Failed cases (${failedCases.length}):
${renderFailedCases(failedCases)}

Current plasmid body:
${plasmid.body}

Identify the likely root cause in the plasmid's ## Behavior or ## Constraints
and propose a MINIMAL edit (1-3 line change) that would address the failures.

Output JSON:
{
  "rationale": "why this edit should help",
  "diff": "unified diff string"
}
```

**Local LLM**: 품질 낮을 수 있음 → `caps.reliableJson=false`면 suggest 기능 **비활성**. 옵션 목록에서 제거.

## 9. Resolved Questions (v0.2)

### Q1 — `[s] Suggest edit` 로컬 LLM 비활성

**확정**: **v0.4 로컬 LLM 환경에서 `[s]` 옵션 완전 비활성 + deterministic 대체 Phase 4+**.

조건: `caps.reliableJson === false` 이면 recovery UI 에서 `[s]` 옵션 제거. 대신 `[i] Inspect` 로 사용자가 패턴 직접 확인하도록.

**Phase 4+ 확장**: `deterministic-suggest-edit.ts` — `preservedConstraints` 위반 감지 기반 규칙 base diff (LLM 없이). 예: "expectation 'output contains X' 실패 → plasmid body 에 X 관련 behavior 누락 추정".

**Rationale**: Local small 모델이 제안 diff 생성은 품질 매우 낮음 (unintended change risk). 사용자에게 빈 결과보다 "이 기능은 cloud LLM 필요" 명시.

### Q2 — Recovery 반복 실패 auto-abort

**확정**: **3회 연속 recovery 실패 시 auto-abort + transcript 보존**.

```typescript
// src/recombination/recovery/strategy.ts
const MAX_RECOVERY_ATTEMPTS = 3;

async function handleRecovery(ctx: RecoveryContext): Promise<RecoveryResult> {
  if (ctx.attemptCount >= MAX_RECOVERY_ATTEMPTS) {
    logAuditEvent({ type: "recovery_auto_abort", ctx });
    throw new RecoveryExhaustedError(ctx);
  }
  // ...
}
```

**Abort 후 상태**: transcript `.partial` 유지, artifact 는 Stage 4 전이면 미저장 (자동 rollback). 사용자에게 `/plasmid inspect recovery-log <sessionId>` 안내.

**Rationale**: Infinite recovery loop 방지. 3회는 empirical 경계 — 대부분 1회 이내 해결, 3회 초과면 구조적 문제 (plasmid schema 또는 LLM capability mismatch).

### Q3 — Auto-fix 적용 후 원본 백업

**확정**: **`.dhelix/plasmids/.autofix-backup/<name>-<timestamp>.md` 자동 저장 (v0.4)**.

7일 retention, 자동 cleanup. 사용자가 auto-fix 원복 필요 시 manual copy.

`/plasmid restore-autofix <name>` (Phase 2+) 편의 커맨드 고려.

**Rationale**: Auto-fix 는 silent change (사용자 재확인 없이 적용). Undo 경로 필수 — 단, cure 와 달리 단순 파일 복원이므로 간단.

### Q4 — `[e] Edit plasmid` 가 여러 plasmid 상호작용 원인인 경우

**확정**: **P-1.16 `ValidationReport.likelyCause` 필드 확장 + recovery 에서 활용**.

```typescript
// src/recombination/validation/reporter.ts (P-1.16 §9)
export interface LikelyCause {
  readonly plasmids: readonly string[];   // 관련 plasmid 이름 (1+)
  readonly category: "single" | "interaction" | "model-capability" | "external";
  readonly evidence: string;
}
```

Recovery UI:
- Category `single` → `[e] Edit <name>` 단일 plasmid
- Category `interaction` → `[e] Edit ... (multiple plasmids may be involved)` + 각각 편집 옵션
- Category `model-capability` → 모델 전환 권장 (`[c] Switch to cloud`)
- Category `external` → DHELIX.md / 환경 변수 확인 안내

**Rationale**: 원인 식별 없이 "[e] Edit" 막연 제시는 사용자 헤매게 함. LikelyCause 분류로 정확한 수정 경로 제시.
