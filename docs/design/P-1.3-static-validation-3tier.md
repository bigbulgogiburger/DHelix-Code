**이 문서의 역할**: Stage 5 (Static Wiring Validation)의 5개 체크를 **severity 3-tier (ERROR/WARN/INFO)** 로 분류하고 각 체크의 실패 대응을 명시.

---

## 0. TL;DR

- 기존 5 카테고리 유지: Reference Integrity, Permission Alignment, Cyclical Dependency, Trigger Conflict, Syntactic Validity
- 3-tier severity: **ERROR** (rollback) / **WARN** (사용자 확인) / **INFO** (참고)
- 각 카테고리 내 세부 check를 severity로 분류 (총 14 check)
- ERROR 1건이라도 있으면 Stage 5 fail → artifact 저장 취소

---

## 1. Severity 3-Tier 정의

| Severity | 의미 | Recombination 영향 |
|---------|-----|------------------|
| **ERROR** | 시스템 무결성 위반, 정상 동작 불가 | Stage 5 fail. Artifact 저장 전 rollback. Cure 후 plasmid 수정 필요. |
| **WARN** | 동작 가능하지만 위험 신호 | Stage 3 Preview에서 사용자에게 표시. 사용자가 `[y]` 승인하면 진행. |
| **INFO** | 참고 정보. 수정 권장 | 기록만. 진행 방해 없음. Report에 표시. |

## 2. 카테고리별 Check 목록 (14 checks)

### 2.1 Reference Integrity (5 checks)

| Check | Severity | 예시 |
|------|---------|-----|
| Agent의 `allowedTools`에 존재하지 않는 tool 참조 | ERROR | `allowedTools: [nonexistent_tool]` |
| Skill의 `triggers`가 빈 배열 | WARN | skill이 작동하지만 trigger 없음 |
| Plasmid `extends: another-plasmid` — 대상 없음 | ERROR | orphan reference |
| Plasmid `compatible-with: X` — X 미존재 | WARN | 표현상 의도 — X가 나중에 추가될 수도 있음 |
| Command가 참조하는 skill 미존재 | ERROR | runtime 실행 불가 |

### 2.2 Permission Alignment (3 checks)

| Check | Severity | 예시 |
|------|---------|-----|
| Agent allowedTools trust level > agent profile trust level | ERROR | low-trust agent가 high-trust tool 호출 |
| Hook이 trustLevel 필드 없음 (default 미적용) | INFO | 기본값 `"safe"` 적용 |
| Skill이 `dangerous` tool 사용하는데 ethical 주석 없음 | WARN | 코드는 동작하지만 governance 미비 |

### 2.3 Cyclical Dependency (2 checks)

| Check | Severity | 비고 |
|------|---------|-----|
| Plasmid A extends B extends A | ERROR | infinite loop |
| Artifact 참조 순환 (agent X → skill Y → agent X) | ERROR | madge-like detector |

### 2.4 Trigger Conflict (2 checks)

| Check | Severity | 비고 |
|------|---------|-----|
| 두 hook이 같은 `(event, matcher)` + 둘 다 blocking | WARN | 실행 순서 결정 필요 |
| 두 skill이 동일 trigger 패턴 | INFO | 사용자가 어느 skill 호출할지 모호 가능 |

### 2.5 Syntactic Validity (2 checks)

| Check | Severity | 비고 |
|------|---------|-----|
| SKILL.md frontmatter Zod schema 위반 | ERROR | skill 로드 실패 |
| Hook TypeScript/bash `-n` 구문 오류 | ERROR | runtime 실행 불가 |

## 3. API

```typescript
// src/recombination/wiring-validator.ts
export type ValidationSeverity = "ERROR" | "WARN" | "INFO";

export interface ValidationFinding {
  readonly category: "reference" | "permission" | "cyclical" | "trigger" | "syntax";
  readonly severity: ValidationSeverity;
  readonly code: string;                   // e.g. "WIRING_REFERENCE_MISSING_TOOL"
  readonly message: string;
  readonly artifact: string;               // 파일 경로
  readonly line?: number;
  readonly remediation?: string;           // 사용자 수정 힌트
}

export interface StaticValidationReport {
  readonly findings: readonly ValidationFinding[];
  readonly errorCount: number;
  readonly warnCount: number;
  readonly infoCount: number;
  readonly passed: boolean;                 // errorCount === 0
}

export async function validateStatic(artifacts: ArtifactSet): Promise<StaticValidationReport>;
```

## 7. Resolved Questions (v0.2)

### Q1 — WARN skip audit

**확정**: **audit log 기록 + Phase 4 correlation 분석**.

`.dhelix/recombination/audit.log` 엔트리: `{ type: "warning_skipped", finding: ValidationFinding, sessionId, timestamp }`. Phase 4 `regression-tracker` 가 skipped warning 과 후속 validation failure 의 **상관관계** 측정 — 2+주 데이터 축적 후 "이 warning은 실질 ERROR로 승격해야 함" 자동 제안.

**Rationale**: Empirical tier upgrade. 처음부터 보수적으로 WARN 분류해도 실사용에서 problematic 확인되면 근거 기반 upgrade.

### Q2 — ERROR auto-fix 자동 적용 vs 제안만

**확정**: **제안만 + `[f] fix` 명시 액션**. Auto-apply 는 `--auto-fix` CLI 플래그 opt-in 한정.

**Rationale**:
- Silent auto-fix는 사용자 expectation 위반 (silent side-effect)
- 제안+명시 선택이 투명
- `--auto-fix` 는 CI 환경 용 (대량 plasmid import 시)

**구현**: `src/recombination/wiring-validator.ts` `autoFixApplicable(finding): SuggestedFix | null`. UX에서 해당 finding 에 `[f]` 옵션 노출.

### Q3 — Permission mismatch 축소 vs 확장

**확정**: **축소 허용 (strict → looser는 안전), 확장은 ERROR**.

```typescript
function isPermissionMismatchError(agent: AgentProfile, tool: Tool): boolean {
  // agent.trustLevel > tool.requiredTrustLevel 면 확장 (ERROR)
  // agent.trustLevel ≤ tool.requiredTrustLevel 면 축소 (OK)
  return trustOrder(agent.trustLevel) > trustOrder(tool.requiredTrustLevel);
}
```

**Rationale**:
- Low-trust agent가 dangerous tool 호출 시도 → ERROR (보안)
- High-trust agent가 safe tool만 사용 → OK (over-qualified but safe)
- 대칭 규칙이면 많은 합법 케이스 막힘
