## 1. 신규 필드 명세

기존 `ModelCapabilities`에 필드 추가. 기존 필드는 모두 유지.

```typescript
export interface ModelCapabilities {
  // ── 기존 필드 (unchanged) ──
  readonly supportsTools: boolean;
  readonly supportsSystemMessage: boolean;
  readonly supportsTemperature: boolean;
  readonly supportsStreaming: boolean;
  readonly maxContextTokens: number;
  readonly maxOutputTokens: number;
  readonly tokenizer: "cl100k" | "o200k" | "llama";
  readonly useDeveloperRole: boolean;
  readonly pricing: ModelPricingInfo;
  readonly useMaxCompletionTokens: boolean;
  readonly capabilityTier: CapabilityTier;
  readonly supportsCaching: boolean;
  readonly supportsThinking: boolean;
  readonly defaultThinkingBudget: number;

  // ── v0.3 신규 (plasmid system) ──
  /** 로컬 LLM 여부 (Ollama, LM Studio, llama.cpp). cloud cascade 결정에 사용. */
  readonly isLocal: boolean;

  /** 모델 파라미터 수 추정 (B 단위, e.g. 8, 32, 70). 없으면 undefined (cloud는 보통 미표기). */
  readonly paramEstimate?: number;

  /**
   * 네이티브 JSON mode 안정성.
   * true: function calling + JSON schema 신뢰 가능 (GPT-4o, Claude, 큰 로컬 모델).
   * false: XML fallback 또는 deterministic fallback 권장 (작은 로컬 모델).
   */
  readonly reliableJson: boolean;

  /**
   * Ollama `format: "json"` request option 지원 여부.
   * true면 sample 프롬프트 없이 JSON 강제 가능.
   */
  readonly supportsFormatJson: boolean;

  /**
   * 모델 고유 digest (Ollama SHA256). Phase 0 preflight에서 probe하여 채움.
   * 이전 recombination과 다르면 model drift 경보.
   * Cloud 모델은 모델 ID (e.g. "claude-opus-4-7") 자체가 digest 역할.
   */
  readonly modelDigest?: string;
}
```

## 2. 기본값 업데이트

```typescript
const DEFAULTS: ModelCapabilities = {
  // 기존 필드 ...
  capabilityTier: "medium",
  // v0.3 신규
  isLocal: false,
  paramEstimate: undefined,
  reliableJson: true,       // 보수적: 큰 cloud 모델 가정
  supportsFormatJson: false,
  modelDigest: undefined,
};
```

**설계 결정**:
- 알려지지 않은 모델이 기본값을 받을 때 `reliableJson: true`로 둬도 안전 — 나중에 interpreter가 parse 실패 시 XML fallback으로 자연스럽게 전환
- `isLocal: false`로 기본값을 두면 모르는 모델은 cloud로 간주 → cascade 정책이 안전측으로 동작

## 3. MODEL_OVERRIDES 확장

### 3.1 기존 cloud 모델 업데이트 (5개)

**Claude 계열 전부 + GPT-4o + GPT-4.1**: `reliableJson: true` 명시 (기존 암묵적 동작을 명시화).

```typescript
// 예: gpt-4o 엔트리
[
  /^gpt-4o/i,
  {
    // ... 기존 필드
    isLocal: false,
    reliableJson: true,
    supportsFormatJson: true,     // OpenAI는 response_format 지원
  },
],
```

변경이 필요한 엔트리: `gpt-4o`, `gpt-4o-mini`, `gpt-4.1`, `gpt-5.*`, `claude-opus-4`, `claude-sonnet-4`, `claude-haiku-4`, `claude-3.5-sonnet`, `o1`, `o3`. 모두 `isLocal: false` + `reliableJson: true`. 단 `supportsFormatJson`은 프로바이더별:
- OpenAI (gpt-*, o-*): `true`
- Anthropic (claude-*): `false` (Anthropic은 tool use 기반이며 format=json 직접 지원 없음)

### 3.2 기존 로컬 패턴 업데이트 (3개)

```typescript
// llama3 기본
[
  /^llama3(?!\.)/i,
  {
    // ... 기존 필드
    isLocal: true,
    paramEstimate: 8,
    reliableJson: false,              // 8B + no function calling
    supportsFormatJson: true,         // Ollama format=json
  },
],

// llama3.1+
[
  /^llama3\.[1-9]/i,
  {
    // ... 기존 필드
    isLocal: true,
    // paramEstimate는 사이즈별 (아래 3.3 Ollama 변형으로 세분화)
    reliableJson: false,
    supportsFormatJson: true,
  },
],

// deepseek-coder (v1)
[
  /^deepseek-coder(?!-v2)/i,
  { /* ... */ isLocal: true, paramEstimate: 6.7, reliableJson: false, supportsFormatJson: true },
],

// deepseek-coder-v2 / v3
[
  /^deepseek-(coder-v2|v3)/i,
  { /* ... */ isLocal: true, paramEstimate: 16, reliableJson: true, supportsFormatJson: true },
],

// codestral, phi, gemma, qwen2.5-coder-7b, qwen2.5-coder-32b
// 모두 isLocal: true 추가 + size 기반 reliableJson 판단 (15B 미만 false)
```

### 3.3 Ollama 사이즈 태그 세분화 (신규 4개)

Ollama는 `llama3.1:8b`, `llama3.1:70b` 처럼 태그로 사이즈 구분. 기존 `/^llama3\.[1-9]/` 패턴은 사이즈 구분 못 함 → **구체적 패턴 먼저** 배치:

```typescript
// llama3.1:70b — 큰 로컬 모델, reliable JSON
[
  /^llama3\.\d+[:-]70b/i,
  { /* ... maxContextTokens 131_072 등 */
    isLocal: true,
    paramEstimate: 70,
    reliableJson: true,
    supportsFormatJson: true,
    capabilityTier: "medium",
  },
],
// llama3.1:8b (기본) — 작은 로컬
[
  /^llama3\.\d+(?::8b)?$/i,
  {
    isLocal: true,
    paramEstimate: 8,
    reliableJson: false,
    supportsFormatJson: true,
    capabilityTier: "low",
  },
],
// qwen2.5-coder:32b — 큰 로컬
[
  /^qwen2\.5-coder[:-]32b/i,
  {
    isLocal: true,
    paramEstimate: 32,
    reliableJson: true,
    supportsFormatJson: true,
    capabilityTier: "medium",
  },
],
// deepseek-coder:6.7b
[
  /^deepseek-coder[:-]6\.7b/i,
  {
    isLocal: true,
    paramEstimate: 6.7,
    reliableJson: false,
    supportsFormatJson: true,
    capabilityTier: "low",
  },
],
```

**순서 주의**: `llama3.1:70b` 패턴이 `llama3.1` 패턴보다 **먼저** 와야 매칭됨 (MODEL_OVERRIDES는 first-match).

## 4. `modelDigest` 채우기 — Runtime Probe

`modelDigest`는 static override에 넣지 않고 Phase 0 preflight (`src/recombination/preflight.ts`)에서 동적으로 채움.

### 4.1 Ollama probe

```typescript
// src/llm/providers/ollama/probe.ts (Phase 1 신규)
export async function probeOllamaModel(modelId: string, baseUrl: string): Promise<string | undefined> {
  try {
    const res = await fetch(`${baseUrl}/api/show`, {
      method: "POST",
      body: JSON.stringify({ name: modelId }),
    });
    if (!res.ok) return undefined;
    const body = await res.json() as { digest?: string };
    return body.digest;   // e.g. "sha256:abc123..."
  } catch {
    return undefined;     // 서버 down 등
  }
}
```

### 4.2 Cloud probe

Cloud 모델은 API가 digest 노출 안 함. 모델 ID 자체를 digest로 사용:

```typescript
export function probeCloudModel(modelId: string): string {
  return `id:${modelId}`;  // e.g. "id:claude-opus-4-7"
}
```

### 4.3 통합

```typescript
export async function enrichCapabilities(modelId: string, provider: ProviderConfig): Promise<ModelCapabilities> {
  const base = getModelCapabilities(modelId);
  const digest = base.isLocal
    ? await probeOllamaModel(modelId, provider.baseUrl)
    : probeCloudModel(modelId);
  return { ...base, modelDigest: digest };
}
```

**주의**: `getModelCapabilities`는 pure (digest 미포함). `enrichCapabilities`는 async + I/O. Phase 1 W1 구현 시 두 함수를 분리.

## 8. Failure Modes

| 실패 | 증상 | 대응 |
|-----|-----|-----|
| Ollama probe 실패 (서버 down) | `modelDigest: undefined` | preflight에서 "drift detection 불가" 경고 후 진행 허용 |
| 새 모델 이름 (e.g. `llama4.0:30b`) | DEFAULTS 적용 (isLocal=false!) | **오분류 리스크** — 이름 패턴에 "로컬 추정" 룰 추가: `(llama\d|qwen|mistral|phi|gemma|deepseek).*:\d+[bm]` → isLocal: true |
| 같은 모델 다른 태그 (`llama3.1` vs `llama3.1:8b`) | 패턴 매칭이 다른 override로 이동 | 위 §3.3 순서 준수 + 각 태그 명시 테스트 |
| `paramEstimate` 없음 | Strategy selector가 `(caps.paramEstimate ?? 0) < 15` 로 처리 → 0이 반환 → "local-small"로 오분류 | `?? 999` 기본값으로 "모르면 보수적으로 큰 모델 취급" — Strategy Selector 설계 (P-1.19)에서 확정 |

## 9. Resolved Questions (v0.2)

### Q1 — `llama3.1` 태그 없음 기본값

**확정**: **태그 생략 시 `8b` 로 추정 (local-small)**.

Ollama 공식 문서: `ollama pull llama3.1` 은 기본적으로 `8b-instruct-q4_K_M` 을 다운로드 (2024-2025 기준). 따라서 `llama3.1` (태그 없음) 은 8B 가정이 현실과 부합.

**MODEL_OVERRIDES 엔트리 순서 재확정**:
```typescript
// 가장 구체적 → 가장 일반
[/^llama3\.\d+[:-]70b/i,  { paramEstimate: 70, ... }],
[/^llama3\.\d+[:-]8b/i,   { paramEstimate: 8, ... }],
[/^llama3\.\d+(?![:-])/i, { paramEstimate: 8, ... }],   // 태그 없음 → 8b
[/^llama3(?!\.)/i,        { paramEstimate: 8, ... }],   // 구버전
```

**Rationale**: Ollama convention (`ollama pull` default tag) 따름. 사용자가 다른 사이즈 원하면 명시 tag.

### Q2 — Cloud `modelDigest` 포맷

**확정**: **v0.4 = `id:<model-id>` 단순 포맷. Phase 3 regression tracker 에서 확장 검토**.

```typescript
export function probeCloudModel(modelId: string): string {
  return `id:${modelId}`;
  // 예: "id:claude-opus-4-7", "id:gpt-5.1-codex"
}
```

**Phase 3+ 확장 고려**: `sha256(systemPromptVersion + modelId + providerVersion)` 으로 확장 → prompt 변경 감지 가능. 하지만 v0.4 에 과도한 복잡성.

**Rationale**: v0.4 수준에서 cloud 모델은 주기적 대규모 업데이트 드물어 단순 ID 충분. Regression tracker (Phase 3) 가 drift 관측 시 확장 trigger.

### Q3 — LM Studio / llama.cpp 탐지

**확정**: **v0.4 scope 밖. Ollama 만 first-class. LM Studio/llama.cpp 는 generic OpenAI-compatible 로 처리**.

**v0.4 동작**:
- Custom base URL 사용 (`OPENAI_BASE_URL=http://localhost:1234/v1`) 이면 `isLocal: true` 로 heuristic 판정
- Heuristic: URL 이 `localhost` / `127.0.0.1` / `0.0.0.0` / `.local` 포함 → local
- `paramEstimate` 는 `undefined` → strategy selector 가 "보수적 local-large" 로 취급

**Phase 2+ 확장**: HTTP response 헤더 분석 (`X-LM-Studio-*`), model list endpoint 형식 차이로 provider 식별.

**Rationale**: Ollama 만 공식 지원 선언 명확성. 나머지는 "작동은 하지만 품질 보장 X" 수준 (generic path).
