# dbcode 전체 기능 목록

> dbcode의 모든 기능을 **기능 / 설명 / 작동방식 / 실행방법** 4열로 정리한 문서입니다.

---

## 1. 내장 도구 (16개)

AI 에이전트가 대화 중 자동으로 호출하는 도구입니다. 사용자는 자연어로 요청하면 AI가 적절한 도구를 선택합니다.

| 기능            | 설명                          | 작동방식                                                                                                                                                                                                                                                                   | 실행방법                                                            |
| --------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| `file_read`     | 파일 내용 읽기                | 텍스트 파일은 줄 번호와 함께 출력. 이미지(PNG/JPG/WebP/SVG)는 Base64 인코딩, PDF는 텍스트 추출(최대 20페이지), Jupyter 노트북(.ipynb)은 셀 단위 포맷. `offset`/`limit`으로 부분 읽기 가능. 2,000줄 자동 절단. 권한: safe                                                   | `"src/index.ts 파일 읽어줘"`, `"README.md 50번째 줄부터 보여줘"`    |
| `file_write`    | 파일 생성/덮어쓰기            | 지정 경로에 파일 작성. 부모 디렉토리가 없으면 `mkdir(recursive: true)`로 자동 생성. 작성 후 줄 수 반환. 타임아웃 30초. 권한: confirm                                                                                                                                       | `"hello.ts 파일을 만들어줘"`, `"이 내용으로 config.json 작성해줘"`  |
| `file_edit`     | 파일 부분 수정                | `old_string`을 찾아 `new_string`으로 교체. 대상 문자열이 파일 내 유일해야 함 (`replace_all: true`로 전체 교체 가능). 수정 후 줄 번호와 ±3줄 컨텍스트 diff 반환. 권한: confirm                                                                                              | `"포트를 3000에서 8080으로 바꿔줘"`, `"이 함수 이름을 변경해줘"`    |
| `bash_exec`     | 셸 명령 실행                  | 플랫폼 감지(bash/zsh/Git Bash/cmd.exe)로 자식 프로세스 생성. stdout/stderr 실시간 스트리밍. 대화형 명령(vim, ssh, git rebase -i) 차단 및 대안 제시. `run_in_background`로 백그라운드 실행 가능. 타임아웃 1초~10분(기본 2분). 권한: confirm                                 | `"npm test 실행해줘"`, `"git log 보여줘"`, `"빌드해줘"`             |
| `bash_output`   | 백그라운드 프로세스 출력 읽기 | `bash_exec`의 `run_in_background`로 실행한 프로세스의 증분 출력 조회. 프로세스 상태(running/exited), 종료 코드, 신규 출력 반환. 권한: safe                                                                                                                                 | `"아까 백그라운드로 돌린 명령 출력 보여줘"` (AI가 자동 호출)        |
| `kill_shell`    | 백그라운드 프로세스 종료      | 실행 중인 백그라운드 프로세스에 시그널(SIGTERM/SIGKILL/SIGINT) 전송. 권한: confirm                                                                                                                                                                                         | `"백그라운드 프로세스 종료해줘"` (AI가 자동 호출)                   |
| `glob_search`   | 파일명 패턴 검색              | `fast-glob` 라이브러리로 glob 패턴 매칭 (예: `**/*.ts`). 결과를 수정 시각(mtime) 기준 최신순 정렬. dotfile 기본 제외. 타임아웃 30초. 권한: safe                                                                                                                            | `"ts 파일 전부 찾아줘"`, `"components 폴더에서 tsx 파일 목록"`      |
| `grep_search`   | 파일 내용 정규식 검색         | ripgrep(`rg`) 우선 사용, 없으면 JavaScript RegExp 폴백. `.gitignore` 준수, 바이너리 자동 스킵. 파일당 최대 200건, 출력 버퍼 10MB. `contextLines`, `caseSensitive`, `multiline` 옵션 지원. 권한: safe                                                                       | `"TODO 주석 전부 찾아줘"`, `"fetchUser 함수 어디서 쓰이는지 검색"`  |
| `list_dir`      | 디렉토리 목록 조회            | 트리 형태(Unicode 커넥터)로 출력. 디렉토리 우선 정렬. `.git`, `node_modules`, `dist`, `build` 등 자동 제외. `recursive`(기본 false), `maxDepth`(1~10, 기본 3) 옵션. 권한: safe                                                                                             | `"src 폴더 구조 보여줘"`, `"프로젝트 디렉토리 구조 알려줘"`         |
| `web_fetch`     | URL 내용 가져오기             | HTTP 요청 후 HTML 정제(script/style/nav 제거, 엔티티 디코딩). 15분 캐시(최대 50 URL). HTTP→HTTPS 자동 업그레이드. 리다이렉트 최대 5회 추적. 콘텐츠 50,000자 제한. 권한: confirm                                                                                            | `"이 URL 내용 가져와: https://..."`, `"공식 문서 읽어줘"`           |
| `web_search`    | 웹 검색                       | Brave Search API(`BRAVE_SEARCH_API_KEY` 설정 시) 또는 DuckDuckGo HTML 파싱 폴백. 제목/URL/스니펫을 마크다운으로 반환. 최대 10건. 타임아웃 10초. 권한: safe                                                                                                                 | `"React 18 useEffect 변경사항 검색해줘"`, `"최신 Node.js LTS 버전"` |
| `notebook_edit` | Jupyter 노트북 셀 편집        | .ipynb JSON 파싱 후 `add`(셀 삽입), `replace`(셀 교체), `delete`(셀 삭제) 동작 수행. 셀 타입: code/markdown. Jupyter 셀 형식에 맞게 소스를 줄 배열로 변환. 권한: confirm                                                                                                   | `"노트북 3번째 셀을 수정해줘"`, `"새 코드 셀 추가해줘"`             |
| `agent`         | 서브에이전트 생성             | 독립된 에이전트 루프를 생성하여 작업 위임. 3가지 타입: `explore`(읽기 전용 탐색), `plan`(구현 계획), `general`(전체 도구 사용). `run_in_background`로 비동기 실행, `isolation: "worktree"`로 Git worktree 격리, `resume`으로 이전 에이전트 이어서 실행 가능. 권한: confirm | `"이 코드베이스 구조 분석해줘"` (AI가 필요 시 자동 생성)            |
| `ask_user`      | 사용자에게 질문               | `ask_user:prompt` 이벤트로 UI에 질문 표시, `ask_user:response` 대기(타임아웃 5분). 선택지 목록(`choices`) 제공 가능. 헤드리스 모드에서는 질문 텍스트를 그대로 반환. 권한: safe                                                                                             | AI가 추가 정보 필요 시 자동 호출. 사용자에게 질문 표시됨            |
| `mkdir`         | 디렉토리 생성                 | `mkdir(recursive: true)` 호출. 이미 존재해도 에러 없음(멱등). 권한: confirm                                                                                                                                                                                                | `"src/utils 폴더 만들어줘"` (AI가 파일 생성 시 자동 호출)           |
| `todo_write`    | 작업 목록 관리                | 인메모리 할일 목록을 상태 아이콘과 함께 관리. 정확히 1개만 `in_progress` 상태여야 함. 매번 전체 목록을 전달해야 함(부분 업데이트 불가). 상태: pending/in_progress/completed. 권한: safe                                                                                    | AI가 복잡한 작업 시 진행 상황 추적용으로 자동 호출                  |

## 2. 슬래시 명령어 (42개)

사용자가 대화 중 `/`로 직접 호출하는 명령어입니다. 39개는 시작 시 자동 등록되며, 3개(`/agents`, `/permissions`, `/team`)는 코드가 존재하지만 아직 메인 앱에 등록되지 않았습니다.

### 2.1 세션 관리

| 기능      | 설명                    | 작동방식                                                                                                                                                | 실행방법                             |
| --------- | ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `/clear`  | 대화 초기화             | 현재 세션의 메시지 히스토리를 전부 비움. 세션 자체는 유지되고 대화 내용만 리셋. `shouldClear: true` 플래그 설정                                         | `/clear`                             |
| `/resume` | 이전 세션 복원          | 인자 없이: 저장된 세션 목록을 대화형 선택 리스트로 표시. 세션 ID 지정 시: 해당 세션의 트랜스크립트를 로드하여 대화 재개. 부분 ID 매칭 지원              | `/resume` 또는 `/resume <세션ID>`    |
| `/fork`   | 세션 분기               | 현재 대화의 모든 메시지를 새 세션으로 복사(`SessionManager.forkSession()`). 분기된 세션은 독립적으로 이어서 사용 가능. `/fork [이름]`으로 이름 지정     | `/fork` 또는 `/fork 새이름`          |
| `/rename` | 세션 이름 변경          | 현재 세션에 사람이 읽을 수 있는 이름을 부여. `/resume` 시 세션 식별을 쉽게 함                                                                           | `/rename 새이름`                     |
| `/export` | 대화 내보내기           | 전체 대화를 마크다운 파일로 저장. 메타데이터 테이블, 턴별 메시지, 도구 호출, 민감정보 마스킹, 통계 요약 포함. `--clipboard` 옵션으로 클립보드 복사 가능 | `/export` 또는 `/export --clipboard` |
| `/copy`   | 코드 블록 클립보드 복사 | 어시스턴트 메시지에서 펜스드 코드 블록을 추출하여 시스템 클립보드(pbcopy/xclip/clip)에 복사. `/copy [N]`으로 특정 블록 번호 지정, 기본은 마지막 블록    | `/copy` 또는 `/copy 2`               |

### 2.2 모델/응답 제어

| 기능            | 설명                  | 작동방식                                                                                                                                                      | 실행방법                                 |
| --------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| `/model`        | AI 모델 변경          | 대화형 선택 리스트로 모델 선택(gpt-4o, gpt-4o-mini, claude-sonnet-4-6, claude-opus-4-6 등) 또는 모델명 직접 지정. 선택한 모델의 캐퍼빌리티 정보도 함께 표시   | `/model` 또는 `/model gpt-4o`            |
| `/effort`       | 추론 깊이 조절        | LLM 응답의 상세함을 4단계로 제어. `low`(1,024토큰), `medium`(2,048), `high`(4,096, 기본), `max`(8,192). 간단한 질문에는 low, 복잡한 분석에는 max 사용         | `/effort low` 또는 `/effort max`         |
| `/fast`         | 빠른 응답 모드 토글   | 일반 모드와 빠른 모드를 전환. 빠른 모드는 더 작고 빠른 모델을 사용하거나 품질을 줄여 응답 속도 향상                                                           | `/fast`                                  |
| `/output-style` | 응답 스타일 변경      | 4가지 스타일: `default`(균형), `explanatory`(이유 포함 상세), `learning`(예시 포함 교육적), `concise`(최소한). 시스템 프롬프트에 스타일 지침 주입             | `/output-style concise`                  |
| `/tone`         | 응답 어조 설정        | 6가지 어조: `normal`(전문적), `cute`(귀여운), `senior`(시니어 개발자), `friend`(친근한), `mentor`(교육적), `minimal`(간결). `refreshInstructions`로 즉시 적용 | `/tone senior` 또는 `/tone cute`         |
| `/architect`    | 설계 모델 설정        | 듀얼 모델 라우팅에서 계획/분석/리뷰에 사용할 고성능 모델 지정                                                                                                 | `/architect claude-opus-4-6`             |
| `/editor`       | 코딩 모델 설정        | 듀얼 모델 라우팅에서 코드 생성에 사용할 비용 효율적 모델 지정                                                                                                 | `/editor gpt-4o-mini`                    |
| `/dual`         | 듀얼 모델 라우팅 토글 | 작업 유형(설계 vs 코딩)에 따라 자동으로 다른 모델을 선택하는 기능 활성화/비활성화                                                                             | `/dual` 또는 `/dual on` 또는 `/dual off` |

### 2.3 컨텍스트/비용 관리

| 기능       | 설명                   | 작동방식                                                                                                                                              | 실행방법                            |
| ---------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| `/compact` | 대화 컨텍스트 압축     | 컨텍스트 윈도우 공간 확보를 위해 수동으로 대화 압축 트리거. 오래된 턴을 구조화된 요약으로 교체. `/compact [주제]`로 특정 주제에 초점을 맞춘 요약 가능 | `/compact` 또는 `/compact 리팩토링` |
| `/context` | 컨텍스트 사용량 시각화 | 모델명, 최대 컨텍스트/출력 토큰 수, 현재 사용량(프로그레스 바 + 퍼센트), 압축 트리거까지 남은 토큰, 역할별 메시지 수 표시                             | `/context`                          |
| `/cost`    | API 비용 상세 분석     | 입력/출력 토큰 수, 추정 비용, 모델별 가격 정보, 효율 지표(턴당 비용, 출력 비율) 표시                                                                  | `/cost`                             |

### 2.4 코드 작업

| 기능        | 설명                | 작동방식                                                                                                                                                                                    | 실행방법                                 |
| ----------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| `/commit`   | Git 커밋 자동 생성  | 스테이지된 diff를 분석하고, 파일 패턴에서 커밋 타입(feat/fix/test/docs) 자동 감지, 최근 커밋 히스토리와 스타일 매칭. LLM에 구조화된 프롬프트를 주입하여 커밋 메시지 생성 및 실행            | `/commit`                                |
| `/review`   | 코드 리뷰           | Git diff를 수집하여 버그, 보안, 코드 품질, 에러 처리를 분석하는 구조화된 리뷰 프롬프트 주입. 이슈를 CRITICAL/HIGH/MEDIUM/LOW로 등급 분류. `--staged` 및 특정 파일 경로 지원. 최대 50KB diff | `/review` 또는 `/review --staged`        |
| `/diff`     | 변경사항 보기       | 스테이지된/안 된 변경사항을 파일별 추가/삭제 라인 수와 함께 표시. `/diff [파일 경로]`로 특정 파일만 조회 가능                                                                               | `/diff` 또는 `/diff src/index.ts`        |
| `/undo`     | 파일 변경 되돌리기  | 수정된 파일 목록을 표시하고, 특정 파일 또는 전체를 마지막 커밋 상태로 복원(`git checkout --`). `/undo [파일 \| all]`                                                                        | `/undo src/index.ts` 또는 `/undo all`    |
| `/rewind`   | 체크포인트 복원     | 자동 생성된 파일 체크포인트 목록(LLM 수정 전 스냅샷)을 파일 수와 diff 요약과 함께 표시. 선택한 체크포인트로 파일 상태 복원                                                                  | `/rewind`                                |
| `/simplify` | 코드 품질 검토      | 변경된 코드에 대해 재사용성(기존 유틸리티), 품질(이름, 구조, 가독성), 효율성(성능, 불필요한 복잡성) 분석 프롬프트 주입                                                                      | `/simplify`                              |
| `/batch`    | 다중 파일 일괄 작업 | glob 패턴과 작업 설명을 받아 매칭되는 모든 파일에 동일 작업을 적용하는 구조화된 프롬프트 생성. `/batch <glob> <작업>`                                                                       | `/batch "src/**/*.ts" 콘솔로그 제거`     |
| `/debug`    | 체계적 디버깅       | 5단계 디버깅 프롬프트 주입: 재현 → 위치 파악 → 분석 → 수정 → 검증. `/debug <에러 설명>`                                                                                                     | `/debug TypeError: Cannot read property` |

### 2.5 설정/진단

| 기능           | 설명               | 작동방식                                                                                                                                                                                                     | 실행방법                                             |

## 3. 커스텀 스킬 (사용자 정의)

마크다운 파일로 재사용 가능한 워크플로우를 정의하는 확장 시스템입니다. dbcode는 **내장 스킬 없이 출시**되며, 사용자가 프로젝트에 맞게 직접 만들어 사용합니다.

### 3.1 스킬 생성 및 사용

| 기능              | 설명                                | 작동방식                                                                                                                         | 실행방법                                            |
| ----------------- | ----------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| 스킬 파일 생성    | 마크다운으로 커스텀 워크플로우 정의 | `.dbcode/skills/<이름>/SKILL.md` 파일에 YAML 프론트매터(name, description) + 프롬프트 본문 작성. 저장 즉시 `/이름` 명령어로 등록 | `.dbcode/skills/my-lint/SKILL.md` 파일 생성         |
| 스킬 호출         | 등록된 스킬을 슬래시 명령으로 실행  | `userInvocable: true`(기본)인 스킬은 `/스킬이름 [인자]`로 호출. 인자는 `$ARGUMENTS`, `$0`~`$N` 변수로 프롬프트에 주입            | `/my-lint src/index.ts`                             |
| 팀 공유 스킬      | 프로젝트 스킬을 Git으로 공유        | `.dbcode/skills/`(프로젝트, 커밋 가능) vs `~/.dbcode/skills/`(사용자 전역, 개인용). 프로젝트 스킬이 우선                         | `.dbcode/skills/`에 스킬 파일 추가 후 `git commit`  |
| 서브에이전트 실행 | 스킬을 독립 에이전트로 실행         | `context: fork` + `agent: explore/plan/general` 설정 시 별도 에이전트에서 실행. 메인 대화 컨텍스트를 오염시키지 않음             | 프론트매터에 `context: fork`, `agent: explore` 설정 |

### 3.2 스킬 파일 예시

```markdown
# .dbcode/skills/code-review/SKILL.md

---

name: code-review
description: 변경된 코드를 리뷰하고 개선점을 제안합니다
argument-hint: "[파일 경로]"

---

## 지시사항

$ARGUMENTS 에 해당하는 파일의 변경사항을 분석하고:

1. 버그 가능성이 있는 코드
2. 보안 취약점
3. 성능 개선 포인트
4. 코드 스타일 위반

각 항목을 CRITICAL/HIGH/MEDIUM/LOW로 분류하여 보고해주세요.
```

위 파일을 저장하면 `/code-review src/utils/path.ts` 명령으로 바로 사용할 수 있습니다.

### 3.3 프론트매터 옵션

| 기능             | 설명                  | 작동방식                                                     | 실행방법                                  |
| ---------------- | --------------------- | ------------------------------------------------------------ | ----------------------------------------- |
| `name`           | 스킬 식별자 (필수)    | 슬래시 명령어 이름으로 사용. `/name`으로 호출                | `name: my-skill`                          |
| `description`    | 스킬 설명 (필수)      | `/help`에 표시되고, AI가 자동 호출 여부 판단에 사용          | `description: 코드를 리뷰합니다`          |
| `argument-hint`  | 사용법 힌트           | `/help`에서 인자 형식 안내                                   | `argument-hint: "[파일 경로]"`            |
| `user-invocable` | 사용자 호출 가능 여부 | `true`(기본): `/name`으로 호출 가능. `false`: AI만 호출 가능 | `user-invocable: true`                    |
| `context`        | 실행 컨텍스트         | `inline`(기본): 현재 대화에 주입. `fork`: 서브에이전트 생성  | `context: fork`                           |
| `agent`          | 서브에이전트 타입     | `context: fork` 시 에이전트 종류 선택                        | `agent: explore` (읽기 전용)              |
| `allowed-tools`  | 도구 화이트리스트     | 이 스킬에서 사용 가능한 도구만 지정                          | `allowed-tools: [file_read, grep_search]` |
| `model`          | 모델 오버라이드       | 이 스킬 실행 시 다른 모델 사용                               | `model: gpt-4o`                           |

## 4. 에이전트 루프 (핵심 엔진)

AI의 사고-행동-관찰 반복 루프입니다.

| 기능                     | 설명                  | 작동방식                                                                                                                                                            | 실행방법                                     |
| ------------------------ | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| ReAct 패턴               | 추론+행동 반복 루프   | LLM 호출 → 도구 호출 추출 → 권한 확인 → 도구 실행 → 결과 추가 → 반복. 종료 조건: 도구 호출 없음(최종 답변), 최대 반복 초과, 사용자 취소(Esc), 서킷 브레이커         | 자동 — 사용자 메시지 전송 시 시작            |
| 최대 반복 제한           | 무한 루프 방지        | 기본 50회(`AGENT_LOOP.maxIterations`). 초과 시 강제 중단. 에이전트 타입별 다름: explore(15), plan(10), general(25)                                                  | 자동 — 반복 횟수 초과 시 자동 중단           |
| 도구 병렬 실행           | 독립 도구 동시 실행   | `groupToolCalls()`로 병렬 가능 여부 판단. file_read/glob/grep: 항상 병렬(읽기 전용). file_write/file_edit: 같은 파일 아니면 병렬. bash_exec: 서로 병렬 가능         | 자동 — AI가 독립 도구 감지 시 병렬 실행      |
| 자동 체크포인트          | 파일 수정 전 스냅샷   | file_write/file_edit 실행 전에 대상 파일의 스냅샷을 자동 생성. `/rewind`로 복원 가능                                                                                | 자동 — 파일 수정 도구 실행 전 자동 생성      |
| 에러 분류 및 재시도      | 에러 유형별 전략      | `transient`(타임아웃, 네트워크) → 지수 백오프 재시도(1s, 2s, 4s). `overload`(429, 503) → 재시도 안 함(클라이언트가 이미 처리). `permanent`(잘못된 요청) → 즉시 실패 | 자동 — 에러 발생 시 유형별 자동 처리         |
| 권한 거부 추적           | 반복 거부 시 중단     | 도구별 거부 횟수를 `Map`으로 추적. 같은 도구 2회 거부 후 LLM에 "이 도구 사용을 중단하라"는 시스템 메시지 주입                                                       | 자동 — 사용자가 도구 권한 2회 거부 시 트리거 |
| MCP 장애 복구            | MCP 도구 실패 시 안내 | `mcp__` 접두사 도구 실패 감지 → 타임아웃 vs 권한 거부를 구분한 복구 가이던스를 시스템 메시지로 주입 → LLM이 실패를 인정하고 대안 제시하도록 유도                    | 자동 — MCP 도구 호출 실패 시 자동 복구       |
| 응답 길이 초과 자동 연장 | 잘린 응답 이어서 생성 | `finishReason === "length"` 감지 시 연속 메시지를 자동 주입하고 루프 재진입                                                                                         | 자동 — 응답이 잘린 경우 자동 연장            |
| 스트리밍                 | 실시간 응답 출력      | 스트리밍 모드 지원. 연결 끊김 시 부분 콘텐츠 복구                                                                                                                   | 자동 — 모델이 스트리밍 지원 시 기본 활성화   |
| 디퍼드 도구 해석         | MCP 도구 지연 로드    | 초기에는 핫 도구만 로드. 마지막 3개 어시스턴트 메시지에서 참조된 MCP 도구의 스키마를 동적 해석                                                                      | 자동 — MCP 도구 참조 시 스키마 자동 로드     |
| 듀얼 모델 라우팅         | 작업별 모델 자동 선택 | `detectPhase()`로 작업 단계 감지(plan/review → 아키텍트 모델, execute → 에디터 모델). 반복마다 적절한 클라이언트/모델로 라우팅                                      | `/dual on`으로 활성화, 이후 자동 라우팅      |
| 결과 절단                | 큰 도구 출력 축소     | 토큰 기반(이진 탐색으로 절단점) 또는 문자 기반(기본 12,000자)으로 도구 결과 축소                                                                                    | 자동 — 도구 출력이 제한 초과 시 자동 절단    |

## 5. 컨텍스트 관리

대화가 길어져도 동작하게 하는 토큰 관리 시스템입니다.

| 기능                    | 설명                                | 작동방식                                                                                                                                                                                                                                                              | 실행방법                                           |
| ----------------------- | ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| 마이크로 압축 (Layer 1) | 대용량 도구 출력을 디스크로 이동    | 매 `prepare()` 호출마다 상시 실행. 200토큰 이상의 도구 출력을 SHA-256 해시로 콜드 스토리지(`~/.dbcode/sessions/{id}/cold-storage/`)에 저장. 최근 5개(HOT_TAIL_SIZE)는 인라인 유지. 우선순위: 에러(100) > 쓰기 결과(80) > 자주 참조(60) > 기타(40) + 최신 보너스(0~10) | 자동 — 매 LLM 호출 전 `prepare()` 시 실행          |
| 자동 압축 (Layer 2)     | 컨텍스트 사용량 임계치 초과 시 요약 | 사용량 ≥ 83.5%(high tier) / 75%(medium) / 65%(low) 도달 시 트리거. LLM 기반 구조화 요약(User Intent, Key Decisions, Files Touched, Errors, Pending Tasks) 또는 로컬 폴백(파일 경로, 에러 발췌, 사용자 메시지 추출). 최근 5/4/3턴은 보존                               | 자동 — 토큰 사용량 임계치 초과 시, 또는 `/compact` |
| 재수화 (Layer 3)        | 압축 후 중요 파일 다시 로드         | 압축 직후 최근/자주 접근한 파일 N개를 컨텍스트에 재로드. 전략: `recency`(최근 접근), `frequency`(최다 접근), `mixed`(최근 3 + 최다 2, 중복 제거). 파일당 4,000자 절단                                                                                                 | 자동 — 압축 완료 직후 자동 실행                    |
| 콜드 스토리지           | 대용량 출력의 디스크 캐시           | 대상 도구: file_read, bash_exec, grep_search, glob_search. 콘텐츠 주소 지정(SHA-256 해시 = 파일명). TTL 24시간. 인라인 콘텐츠를 `[cold-ref:{hash}]` 참조로 교체                                                                                                       | 자동 — 마이크로 압축 시 자동 저장                  |
| 적응적 GC               | 사용량에 따른 정리 간격             | 사용량 > 80%: 매번 GC. 50~80%: 5회마다. < 50%: 15회마다                                                                                                                                                                                                               | 자동 — 사용량 수준에 따라 주기적 실행              |

## 10. 복구 시스템
