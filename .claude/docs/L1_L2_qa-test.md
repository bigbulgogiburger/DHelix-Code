## 주요 발견사항 (Key Findings)

### 1. 헤드리스 멀티턴 한계 (구조적)

**발견**: `-p` 헤드리스 모드에서 `--resume SESSION_ID`를 함께 사용해도 진짜 대화 컨텍스트가 유지되지 않는다.

**원인**: `src/index.ts`에서 `-p` 플래그 감지 시 `runHeadless()`를 호출하고 **세션 관리 코드 전에** early return. 따라서 매 실행이 독립적인 새 세션으로 시작됨.

**영향**: TC-25(10턴 컨텍스트 유지)는 헤드리스 모드에서 불가. 나머지 멀티턴 TC들은 파일시스템 기반 컨텍스트로 우회 가능.

**권장**: 헤드리스 모드에서 진짜 멀티턴을 원한다면 `--conversation-file` 등 별도 구현 필요.

### 2. 가드레일 False Positive

**발견**: 여러 합법적인 프롬프트가 "I cannot assist with that request"로 거부됨:

- "프로젝트를 실행하면 타입 에러가 나" (TC-16)
- "tsc를 실행해줘" (TC-26)
- "의존성을 grep으로 검색해서 미사용 패키지 목록을 알려줘" (TC-28)
- "Console.log를 사용하는 파일을 모두 찾아줘" (TC-10 초회)

**영향**: TC-16 -2점, TC-26 -3점, TC-28 완전 실패의 주요 원인.

**권장**: 가드레일 프롬프트 패턴 재검토. 코드 분석/타입 체킹 요청은 거부 목록에서 제외 필요.

### 3. API 타임아웃 (60초)

**발견**: dbcode 내부 API 타임아웃 60초 (`src/config/defaults.ts: timeout: 60_000`)가 Bash 실행 타임아웃(120초)보다 짧아 복잡한 태스크에서 EXIT: 124 발생.

**영향**: TC-17 T4, TC-18 T2/T3, TC-19 부분적 실패.

**권장**: E2E 테스트 시 간단하고 구체적인 프롬프트 사용. 복잡한 작업은 여러 턴으로 분할.

### 4. 파일/코드 검색 범위 제한

**발견**:

- TC-04: `src/` 파일 카운트 시 하위 디렉토리 포함해서 과다 계산
- TC-07: `console.log` 검색 시 `src/` 내부를 누락하고 루트 파일만 검색
- TC-12: glob 파일 카운트 오차 6개 (±5 초과)

**영향**: 파일시스템 탐색/집계 태스크의 정확도가 낮음.

**권장**: 검색 범위를 명시적으로 지정하는 프롬프트 작성 필요.

### 5. Auto-answer 오작동

**발견**: `[Question]` 인터랙션에서 `[Auto-answer]`가 적절하지 않은 응답을 제공:

- TC-14: 프레임워크 질문 시 "Jest" (프로젝트는 Vitest 사용)
- TC-22: Calculator 설계 방향 선택에서 부적절한 자동 응답

**영향**: 테스트 프레임워크 불일치, 잘못된 구현 방향.

**권장**: E2E QA 시 auto-answer 로직 개선 또는 명시적 응답 없이 인터랙션 제거 프롬프트 사용.

### 6. 의존성 분석 실패 (TC-28)

**발견**: "미사용 패키지 찾기" 태스크에서 여러 프롬프트가 blank output 또는 guardrail 거부. dbcode가 자신의 소스코드를 분석하는 자기참조적 태스크에서 특히 취약.

**영향**: TC-28 3점 (10/17 패키지 정확, 6/17 false positive). grep 기반 분석이 re-export 및 간접 import 패턴을 놓침.

**권장**: 자기참조 분석(dbcode가 자신의 package.json을 분석)은 다른 디렉토리에서 실행하거나 프롬프트를 충분히 구체화. 패키지명 명시적 열거 시 분석 성공률 향상.
